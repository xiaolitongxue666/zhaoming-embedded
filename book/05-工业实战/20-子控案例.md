# 第 20 章 · 工业控制板子控案例

参考代码：[`industrial/motor_24vfuncs/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/motor_24vfuncs/) 是水平电机 + 垂直电机驱动的脱敏骨架，[`industrial/platform_layer/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/platform_layer/) 是 UART / I²C / SPI 统一抽象层。两份骨架跟章节正文里贴的接口字节级对应。完整可 build 的小工程参考附录 B / C 的 [`industrial/stm32_full/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/stm32_full/) 和 [`industrial/linux_full/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/linux_full/)。

上一章讲主控板，它管的是相对简单的外设：开关型的 LED、读写型的 EEPROM、读温度的传感器、状态机带回调的风扇。这一章讲子控板，它管的是真正复杂的事：两路电机的运动控制。

电机驱动是工业项目里 OOP 抽象能力最被考验的地方。原因有三个：

第一，电机品种多。同一块板可能既要驱动伺服电机（通过 UART 走厂家私有命令协议），又要驱动直流电机（H 桥 PWM 控制）。两种电机能力差距巨大但应用层接口必须统一。

第二，电机的控制接口不是"开"和"关"两个动作，是 24 个独立动作（启动、停止、速度模式、相对移动、绝对定位、回零、加减速度、限位、跟随误差、I²t 保护 ...）。

第三，电机有大量异步事件（位置变化、状态变化、错误中断），必须支持多回调。

子控板项目里水平电机驱动 `h_motor` 把这三件事一次性解决。下面围绕它展开整章，再把垂直电机 `v_motor` 作为对照，把 Platform 抽象层 UART/I²C/SPI 三栈看一遍，最后给出跨 MCU 移植清单。

## 20.1 24 个虚方法的 ops 表

节选自子控板项目 `app/drivers/h_motor/h_motor_base.h`：

```c
typedef struct {
	void (*motor_stop)(struct h_motor_base *me);
	void (*motor_start)(struct h_motor_base *me);
	void (*move_velocity_mode)(struct h_motor_base *me,
	                            h_motor_base_move_dir_t dir);
	void (*move_relative_mode)(struct h_motor_base *me, double move_mm);
	void (*move_absolute_mode)(struct h_motor_base *me, double pos_mm);
	void (*velocity_set)(struct h_motor_base *me, int32_t velocity);
	void (*acce_dece_set)(struct h_motor_base *me,
	                      uint32_t acce, uint32_t dece);
	void (*manual_mode)(struct h_motor_base *me, bool is_manual);
	void (*status_get)(struct h_motor_base *me);
	bool (*status_get_sync)(struct h_motor_base *me,
	                        h_motor_base_status_t *status);
	void (*cur_pos_get)(struct h_motor_base *me);
	void (*enable)(struct h_motor_base *me, bool enable);
	void (*home)(struct h_motor_base *me);
	void (*home_set)(struct h_motor_base *me, double offset_mm);
	void (*positive_limit)(struct h_motor_base *me, uint32_t limit_mm);
	void (*negative_limit)(struct h_motor_base *me, int32_t limit_mm);
	void (*fault_clear)(struct h_motor_base *me);
	void (*following_err_enable)(struct h_motor_base *me, bool enable);
	void (*following_err_limit_set)(struct h_motor_base *me,
	                                double limit_mm);
	void (*encoder_ratio_set)(struct h_motor_base *me, double cnt_per_mm);
	void (*i2t_fault_enable)(struct h_motor_base *me, bool enable);
	void (*i2t_peak_current_set)(struct h_motor_base *me,
	                              uint16_t current_001a);
	void (*i2t_continuous_current_set)(struct h_motor_base *me,
	                                   uint16_t current_001a);
	void (*i2t_peak_time_set)(struct h_motor_base *me,
	                           uint16_t time_ms);
} h_motor_base_ops_t;
```

24 个函数指针。这是这本书里你看到的最大的 ops 表。

逐组看这些方法在解决什么问题：

| 组 | 方法 | 解决什么 |
|---|---|---|
| 启停 | `motor_start / motor_stop / enable` | 上电、断电、安全停止 |
| 三种运动模式 | `move_velocity_mode / move_relative_mode / move_absolute_mode` | 速度模式（持续转动）、相对移动（再走 100mm）、绝对定位（去到 500mm） |
| 运动参数 | `velocity_set / acce_dece_set` | 速度、加减速度 |
| 状态查询 | `status_get / status_get_sync / cur_pos_get` | 异步触发 + 同步阻塞 + 读位置 |
| 回零 | `home / home_set` | 寻找零点开关、设置零点偏移 |
| 限位 | `positive_limit / negative_limit` | 软件限位 |
| 故障 | `fault_clear` | 清除故障状态 |
| 跟随误差 | `following_err_enable / following_err_limit_set` | 实际位置和命令位置差超过阈值时报错 |
| 编码器 | `encoder_ratio_set` | 编码器脉冲数 / 毫米的比例 |
| I²t 保护 | `i2t_fault_enable / i2t_peak_current_set / i2t_continuous_current_set / i2t_peak_time_set` | 电机过载保护 |

这些方法不是凭空想出来的。它们是真正的工业控制板对电机驱动的全部需求。在 ACM 伺服电机上，每个虚方法对应一次"把命令打包成厂家私有协议帧、推进工作队列、由工作线程异步走 UART 发出去"的过程：

```c
/* 节选自 h_motor_acm.c 内部，简化示意 */
typedef struct {
	uint8_t  cmd;       /* 厂家私有命令字节, 例如速度 / 位置 / I2T 限流 */
	int32_t  arg;
} acm_work_t;

static void acm_velocity_set(h_motor_base_t *me, int32_t velocity)
{
	h_motor_acm_t *self = (h_motor_acm_t *)me;
	acm_work_t work = {
		.cmd = ACM_CMD_VELOCITY_SET,
		.arg = velocity,
	};
	osMessageQueuePut(self->work_queue, &work, 0, 0);
}

static void acm_following_err_limit_set(h_motor_base_t *me, double limit_mm)
{
	h_motor_acm_t *self = (h_motor_acm_t *)me;
	int32_t limit_cnt = (int32_t)(limit_mm * self->encoder_cnt_per_mm);
	acm_work_t work = {
		.cmd = ACM_CMD_FOLLOWING_ERR_SET,
		.arg = limit_cnt,
	};
	osMessageQueuePut(self->work_queue, &work, 0, 0);
}

static void acm_i2t_peak_current_set(h_motor_base_t *me,
                                     uint16_t current_001a)
{
	h_motor_acm_t *self = (h_motor_acm_t *)me;
	acm_work_t work = {
		.cmd = ACM_CMD_I2T_PEAK_SET,
		.arg = current_001a,
	};
	osMessageQueuePut(self->work_queue, &work, 0, 0);
}
```

每个虚方法只做一件事：把命令打包推进工作队列，立刻返回。真正的串口字节流由 `acm_work_thread` 内部按厂家协议组帧后从 UART 发出。这就是"代码长什么样的"在工业项目里的最终形态：应用层调一个 `h_motor_velocity_set(h_motor_dev, 200)`，进到基类层封装函数内部走 `me->ops->velocity_set(me, 200)` ops dispatch，跳转到 `acm_velocity_set`，把命令推进工作队列就返回。工作线程异步把命令组帧、走 UART 总线、收响应、必要时回调通知应用层。应用层永远只见普通函数 `h_motor_velocity_set(...)`，看不到 ops 字段、看不到工作队列、看不到 UART。这一连串操作书前面 18 章每一章都铺垫过。

24 个 ops 在基类层并不是同一种约束。看一眼仓库里的 `h_motor_base.c`：核心动作（`motor_stop / motor_start / move_velocity_mode / move_relative_mode / move_absolute_mode / velocity_set / acce_dece_set / status_get / cur_pos_get / enable / home / home_set / positive_limit / fault_clear / manual_mode`）是**纯虚**——基类封装函数派发前先 `platform_assert(me->ops->xxx != NULL)`，子类构造函数忘了挂钩上电立刻 assert fail 把低级 bug 砸到台面上。剩下一组（`negative_limit / following_err_enable / following_err_limit_set / encoder_ratio_set / i2t_*`）是**可空**——基类用 `if (me->ops->xxx) me->ops->xxx(me, ...);` 兜底，老固件子类如果没这能力直接留 NULL，应用层调一下就 no-op 不会崩。`status_get_sync` 还有第三种处理：状态机关键路径上必须不会崩，所以走完整 NULL 校验后返回 `false`，让上层有机会按"读不到"分支降级。一张 ops 表里同时存在三种约束级别——纯虚 / 可空 / 软失败——每一种都对应一类真实工程权衡。

### 20.1.1 24 个虚方法是怎么长出来的

第一次读到 24 个虚方法你可能本能地觉得"太多了"。这一节讲清楚：每个虚方法都是工程演化逼出来的，没有一个是装饰。

最早的版本只有 4 个方法：`start / stop / move / enable`。这能驱动玩具电机，跑工业场景一周就崩。下面这张表对照着写出哪个方法是因为踩了什么坑加上来的：

| 加上的方法 | 对应的工程问题 |
|---|---|
| `move_velocity_mode` / `move_relative_mode` / `move_absolute_mode` | 早期 `move(速度, 距离)` 一个接口表达不清三种语义。"持续转 vs 走指定距离 vs 去指定坐标"必须分开 |
| `velocity_set` 单独抽出 | 运动中要动态改速度，把 velocity 从 move 参数里抽出来变成独立 setter |
| `acce_dece_set` | 加速度过大伺服报警，必须能配加减速曲线 |
| `home` / `home_set` | 上电不知道电机当前位置，必须找零点开关；零点不是绝对零，要支持偏移 |
| `positive_limit / negative_limit` | 软件限位，超出范围拒绝运动 |
| `following_err_enable / following_err_limit_set` | 跟随误差监控（命令位置 vs 实际位置差），是伺服系统标配的安全机制 |
| `i2t_fault_enable / i2t_peak_current_set / i2t_continuous_current_set / i2t_peak_time_set` | I²t 过载保护：短时大电流 + 长时小电流的电机损坏阈值 |
| `status_get` / `status_get_sync` 两份 | 异步状态查询（不阻塞）和同步状态查询（必须立刻拿到）业务上都要 |
| `cur_pos_get` 异步 | 位置读取频率高，避免阻塞业务线程 |
| `manual_mode` | 上位机要切到手动操作模式，电机要从自动控制切到外部按键控制 |
| `fault_clear` | 故障锁定后要主动清除才能继续动作 |
| `encoder_ratio_set` | 不同机械传动比，编码器脉冲数和实际位移之间要标定 |

每加一行 ops 都来自实际现场的需求或事故。这就是工业接口为什么"看起来比教学版臃肿"的原因。

## 20.2 子类 h_motor_acm：把异步操作翻译成阻塞接口

电机驱动里最棘手的不是 24 个虚方法本身，而是它们底下都是**异步**操作。

伺服电机通过 UART 走厂家私有协议下命令。下完命令到电机响应，中间有几毫秒的等待。期间应用层不能干等。但应用层调用 `h_motor_velocity_set(h_motor_dev, 200)` 这一行写起来必须像同步的（基类层封装函数内部走 `me->ops->velocity_set(me, 200)` 把命令推进队列）。

工业代码的解法是：每个电机实例自带一个工作线程加消息队列。所有 ops 调用先把命令推进队列，立刻返回；工作线程把队列里的命令一条条取出来，按厂家协议组帧、从 UART 发出去、等响应，必要时通过回调通知应用层。

节选自 `app/drivers/h_motor/h_motor_acm.h`：

```c
typedef struct {
	h_motor_base_t base;      /* 基类放在第一字段 */

	platform_device_t  uart_dev;
	int32_t            clutch_pin;
	int32_t            rev_en_pin;
	int32_t            en_pin;
	int32_t            fwd_pin;
	int32_t            fault_detect_pin;

	osMessageQueueId_t work_queue;     /* 命令队列 */
	osMessageQueueId_t rx_queue;
	osThreadId_t       work_thread;    /* 工作线程 */
	osThreadId_t       rx_thread;

	h_motor_acm_rx_buf_t rx_buf;
	uint32_t             rcv_last_byte_ticks;

	Button                  fault_detect_btn;
	h_motor_base_status_t   status;

	double  encoder_cnt_per_mm;
	int32_t max_following_error;
	int16_t max_motor_current;
	int16_t max_i2t_sum;
	bool    i2t_reg_unsupported;
	uint8_t debug_print_cnt;
	uint32_t last_work_id;
} h_motor_acm_t;
```

注意几个工程实践的细节。

**`base` 字段放第一位。** 子类 struct 的第一个字段是基类对象 `h_motor_base_t base`，向上转型零代价（ch12 已讲透）。命名上整个项目（教学版的 `led_basic` / `v_motor`、工业版的 `h_motor`）统一叫 `base`，避免读者读到这一段时还要在脑里切换"哦这块叫 super 那块叫 base"——Linux 内核里也常见 `dev` / `parent` / `super` 等变体写法，叫什么不重要，**一个项目内部一致**才是工程纪律。

**两个工作线程加两个消息队列。** `work_queue + work_thread` 处理"应用层下来的命令"，`rx_queue + rx_thread` 处理"电机串口响应数据"。两套独立避免死锁（命令线程在等响应时不会卡死接收线程）。

**fault_detect_btn。** 故障检测引脚被当成"按键"对待。`Button` 来自 MultiButton 库，专门做去抖动加长按 / 短按 / 双击事件。把硬件故障引脚的电平变化也当作"按键事件"统一处理。这是教学版从来想不到的细节。

**最大值字段（max_following_error / max_motor_current / max_i2t_sum）。** 运行期间持续记录峰值。出问题时这些字段是黄金调试信息，能告诉你"电机最大跟随误差到多少"、"瞬时电流多大"。教学版不会写这种东西，工业版离了就是黑盒。

**`i2t_reg_unsupported` 标志位。** 适配同型号但不同固件版本的伺服电机：旧固件没有 I²t 寄存器，标志位置 `true` 后 ops 里相关函数变 no-op。教学版常忽略"同一颗芯片不同固件"这种工业现实。

**`last_work_id`。** 命令排序号。把命令推进 work_queue 时打一个 id，处理完一条响应一条。配合 `debug_print_cnt` 调试时能精确知道"哪条命令响应丢了"。

构造函数（节选）：

```c
typedef struct {
	const char *uart_name;
	const char *clutch_pin_name;
	const char *rev_en_pin_name;
	const char *en_pin_name;
	const char *fwd_pin_name;
	const char *fault_detect_pin_name;
} h_motor_acm_init_param_t;

void h_motor_acm_init(h_motor_acm_t *me,
                      h_motor_acm_init_param_t *init_param);
```

构造参数全是字符串引脚名（"PE.0"、"PE.1" 等）。具体引脚号通过 `platform_pin_get(name)` 查表得到。这是 platform 抽象的核心：上层代码不写"PE_0 = 0x4801080C"，只写字符串名字。换芯片只改 platform 层的引脚名表。

## 20.3 v_motor 子类：同一个基类指针指两种完全不同的电机

子控板上还有一路垂直电机。它和水平电机硬件方案完全不同：水平电机是 ACM 伺服电机（UART + 厂家私有命令协议），垂直电机是 IFX007T H 桥 PWM 控制的直流电机。控制方式天差地别。

垂直电机基类（节选自 `app/drivers/v_motor/v_motor_base.h`）：

```c
typedef enum {
	V_MOTOR_BASE_MOVE_DOWN = 0,
	V_MOTOR_BASE_MOVE_UP,
	V_MOTOR_BASE_MOVE_STOP,
} v_motor_base_dir_t;

typedef struct {
	void (*motor_stop)(struct v_motor_base *me);
	void (*motor_move)(struct v_motor_base *me,
	                   v_motor_base_dir_t dir);
	void (*fault_clear)(struct v_motor_base *me);
} v_motor_base_ops_t;

typedef struct v_motor_base {
	const v_motor_base_ops_t *ops;
	v_motor_base_dir_t move_state;
	v_motor_move_status_changed v_motor_move_status_changed_cb;
	v_motor_err_cb_t err_cb;
} v_motor_base_t;
```

只有 3 个 ops 方法（停 / 走 / 清故障），加 2 个回调（状态变化 / 错误）。比水平电机简单一个量级。原因是直流电机没有"绝对定位"的概念，能力域窄得多。

IFX007T 子类的教学最小内核（节选自 `v_motor_ifx007t.h`）：

```c
typedef struct {
	v_motor_base_t base;       /* 基类放在第一字段 */
	int32_t ctrl_1_pin;
	int32_t ctrl_2_pin;
	int32_t sleep1_pin;
	int32_t sleep2_pin;
} v_motor_ifx007t_t;
```

IFX007T 是英飞凌一颗 H 桥驱动芯片。两个控制引脚 + 两个使能引脚。`motor_move(UP)` 在子类里就是把 `ctrl_1_pin` 拉高、`ctrl_2_pin` 拉低、两个 sleep 拉高使能；`motor_move(DOWN)` 反过来；`motor_move(STOP)` 把两个 ctrl 都拉低。简单到几行 GPIO 操作就完成。

> 仓库里 `v_motor_ifx007t.h` 是产品级版本，子类 struct 字段更多（两路 PWM 设备 + 两个减速区限位开关 + 一路 IS 电流采样 ADC + 工作线程 + 周期定时器 + 软启动 / 加减速 / 故障检测的运行时状态），上面这块代码是把它压成"OOP 抽象骨架"的教学最小核。产品代码里 `ctrl_1` / `ctrl_2` 走的是 PWM 调速而不是 GPIO 推挽，但抽象层面就是控制 H 桥两路高边臂——子类内部具体怎么斩波怎么爬坡，基类层一字看不见。完整产品级实现在 § 20.10.2 里展开。

应用层接口风格完全统一：

```c
/* environment_cfg/environment_export.h */
extern h_motor_base_t *h_motor_dev;
extern v_motor_base_t *v_motor_dev;     /* 同样是 *_base 类型 */

/* 应用层调封装函数，看不到 ops 表 */
h_motor_move_absolute(h_motor_dev, 1500.0);
v_motor_move(v_motor_dev, V_MOTOR_BASE_MOVE_UP);
```

`h_motor_move_absolute` / `v_motor_move` 是 `h_motor.c` / `v_motor.c` 里的封装函数（同 ch01 1.10 节 `led_on` 套路），内部走 `me->ops->move_absolute_mode(me, ...)` / `me->ops->motor_move(me, ...)` dispatch 到具体电机实现（ACM 伺服 / IFX007T H 桥）。

垂直电机是另一组 base + ops 表（接口比水平电机简单，因为直流电机的能力域窄）。两个基类不互相继承，是平行的。

这反映一个 OOP 设计原则：**继承沿"能力 / 接口的相似度"建，不沿"硬件的相似度"建**。水平电机和垂直电机在硬件层面相似度最低，但因为它们都得"被应用层用同一种方式驱动",所以各自有一个独立的 base 接口。教学版常犯的错是看到"这两个东西都是电机"就硬塞到一个基类里，最后发现各种"水平电机的方法垂直电机不需要"、"垂直电机的方法水平电机不支持"，被迫加 `if (motor_type == HORIZONTAL) ... else ...`，多态退化成手动派发。

子控板项目这一招让水平电机的 24 虚方法和垂直电机的 3 虚方法各自独立演化，互不污染。某一天垂直电机上 PWM 调速能力，理论上只在 `v_motor_base_ops_t` 加一行 `void (*speed_set)(...)`，水平电机一行不动。这是"把变化关在它原本属于的位置"。单一职责原则在 driver 层级的体现。

实际仓库里产品级 `v_motor_ifx007t.c` 走的是另一条路：根本不在 ops 表新加 `speed_set`，而是直接把 PWM 软启动 + 加减速曲线塞进 `motor_move` 内部由工作线程异步跑出来。原因是直流电机的速度策略（起步占空多少、ramp 多长、什么时候踩限位降速）本身是机械相关硬编码的——上层应用层永远只说"往上走"或"停"，不需要也不应该参数化"以多少占空走"。两条路都合法：能力域真的扩出来一档独立 setter，就加 ops；能力还在"开 / 关 / 方向"原本三个虚方法的语义内，就吸收进子类内部。设计判断的依据始终是**应用层视角能力是否真的多一档**。

## 20.4 三种回调：状态、位置、错误

电机异步地往应用层喷三种事件：状态变化、位置变化、错误。每种独立一个回调。

```c
typedef void (*status_cb_t)(h_motor_base_status_t *status,
                            struct h_motor_base *me);
typedef void (*cur_pos_cb_t)(double pos_mm,
                             struct h_motor_base *me);
typedef void (*err_cb_t)(h_motor_base_err_id_t err_id,
                         struct h_motor_base *me);

typedef struct h_motor_base {
	h_motor_base_ops_t *ops;
	status_cb_t  status_cb;
	cur_pos_cb_t cur_pos_cb;
	err_cb_t     err_cb;
} h_motor_base_t;
```

应用层注册：

```c
h_motor_base_status_get_cb_register(h_motor_dev, on_motor_status);
h_motor_base_cur_pos_cb_register(h_motor_dev, on_motor_pos_update);
h_motor_base_fault_cb_register(h_motor_dev, on_motor_error);
```

注册之后，电机驱动内部凡是状态有变化就调 `me->status_cb(...)`，位置每 10ms 上来一次就调 `me->cur_pos_cb(...)`，发生通信故障 / 硬件故障就调 `me->err_cb(...)`。

为什么不像主控板上某些驱动那样有 5 种 cb？因为电机的事件类型确实就这三种。每种事件类型一个 cb，做到"刚好够用、不再多 1 个、不再少 1 个"。OOP 不是越多越好，是匹配真实需求。

回调里传基类指针 `struct h_motor_base *me` 让回调函数知道"我是为哪个电机被调起来"。系统里有水平电机一路 + 垂直电机一路时，同一个 `on_motor_status` 函数能服务两个 motor，靠 `me` 区分。这就是教学版 ch01 讲过的"挂号单"在事件回调场景的延伸。

## 20.5 编码器与位置开关：两个简单驱动作对照

不是所有驱动都需要 ops 表。看两个反面例子：编码器（encoder）和位置开关（position_switch）。

**编码器。** 节选自 `app/drivers/encoder/encoder.h`：

```c
struct encoder;

typedef void (*encoder_cur_pos_get_cb_t)(struct encoder *me, double cur_pos);
typedef void (*encoder_fw_version_cb_t)(struct encoder *me, uint32_t ver);
typedef void (*encoder_glitch_cnt_cb_t)(struct encoder *me, uint32_t cnt);

typedef struct encoder {
	platform_spi_device_t  *spi_dev;
	int32_t                 overspeed_pin;
	int32_t                 reserved_pin0;
	int32_t                 reserved_pin1;
	encoder_cur_pos_get_cb_t cur_pos_get_cb;
	encoder_fw_version_cb_t  fw_ver_cb;
	encoder_glitch_cnt_cb_t  glitch_cnt_cb;
	osMessageQueueId_t       work_queue;
	osThreadId_t             work_thread;
} encoder_t;

void encoder_init(encoder_t *me, encoder_init_param_t *init_param);
void encoder_cur_pos_get(encoder_t *me);
void encoder_cur_pos_cb_register(encoder_t *me,
                                 encoder_cur_pos_get_cb_t cb);
```

只一种实现（项目里就只有一种 SPI 编码器），没有 ops 表。但仍然是 struct + me + 三种回调 + 工作线程的完整结构。

**位置开关。** 节选自 `app/drivers/position_switch/position_switch.h`：

```c
typedef struct position_switch {
	int32_t top_stop_pin;
	int32_t top_dece_pin;
	int32_t bottom_stop_pin;
	int32_t bottom_dece_pin;
	int32_t home_pin;
	position_swich_change_cb_t status_change_cb;
	Button top_stop_btn;
	Button top_dece_btn;
	Button bottom_stop_btn;
	Button bottom_dece_btn;
	Button home_btn;
} position_switch_t;
```

5 个限位开关塞进一个 struct，每个开关用 `Button` 库做去抖。一个统一的 `status_change_cb` 把哪个开关变化了通过 enum 参数告诉上层：

```c
typedef enum {
	POSITION_SWITCH_TOP_STOP = 0,
	POSITION_SWITCH_TOP_DECE,
	POSITION_SWITCH_BOTTOM_STOP,
	POSITION_SWITCH_BOTTOM_DECE,
	POSITION_SWITCH_HOME,
} position_switch_id_t;

typedef void (*position_swich_change_cb_t)(position_switch_id_t id,
                                            position_switch_status_t status,
                                            struct position_switch *me);
```

教学版会犯的错：5 个开关写 5 份 driver 代码。或者反过来：把 5 个开关塞到 5 个 ops 函数里。两个都不对。这里的解法是"概念上属于同一个东西的多个开关"打包成一个 struct，事件类型用 enum 区分。

## 20.6 Platform 层：UART / I²C / SPI 的统一抽象

子控板这边的电机控制要走 UART（RS-485 物理层 + 厂家私有命令协议），编码器要走 SPI，温度传感器要走 I²C。三种总线接口完全不同，但应用层不应该感知到这些差别。Platform 层负责吸收这些差别。

`app/platform/` 目录下的核心头文件：

```
platform_def.h           # 错误码、通用类型
platform_pin.h           # GPIO 抽象
platform_uart.h          # UART 抽象
platform_i2c.h           # I²C 抽象
platform_spi.h           # SPI 抽象
platform_device.h        # 设备注册表
platform_module_export.h # initcall 宏（上一章讲过）
arch/cortex_m7/...       # ARM Cortex-M7 实现
board/h7_evb/...         # 这块板子的引脚映射
```

UART 抽象的接口（节选自 `platform_uart.h`）：

```c
typedef struct platform_uart_configure {
	uint32_t baud_rate;
	uint32_t data_bits   :4;
	uint32_t stop_bits   :2;
	uint32_t parity      :2;
	uint32_t bit_order   :1;
	uint32_t invert      :1;
	uint32_t flowcontrol :1;
	uint32_t reserved    :5;
} platform_uart_configure_t;

typedef struct platform_uart_device {
	struct platform_device parent;             /* 嵌入通用 device 基类 */
	const struct platform_uart_ops *uart_ops;
	platform_uart_configure_t configure;
} platform_uart_dev_t;

typedef struct platform_uart_ops {
	platform_err_t (*open)(platform_uart_dev_t *dev);
	platform_err_t (*close)(platform_uart_dev_t *dev);
	uint32_t (*read)(platform_uart_dev_t *dev,
	                 uint32_t pos, void *pbuf, uint32_t size);
	uint32_t (*write)(platform_uart_dev_t *dev,
	                  uint32_t pos, const void *pbuf, uint32_t size);
	platform_err_t (*configure)(platform_uart_dev_t *dev,
	                            platform_uart_configure_t *cfg);
} platform_uart_ops_t;

platform_err_t platform_hw_uart_register(platform_uart_dev_t *dev,
                                         const char *name, void *pdata);
```

注意第一个字段 `struct platform_device parent`：这是**两层继承**。`platform_uart_dev_t` 继承通用 `platform_device`（提供 register / find / open / close 这些通用接口），自己再加 UART 特有的 `uart_ops` 和 `configure`。这就是 ch12 讲过的"基类放第一个字段"在 platform 层的应用。

应用层（或者驱动层）调用：

```c
platform_uart_dev_t *uart3 = (platform_uart_dev_t *)
                             platform_device_find("uart3");
uart3->uart_ops->configure(uart3, &my_cfg);
uart3->uart_ops->open(uart3);
uart3->uart_ops->write(uart3, 0, data, len);
```

完全是第 11 章的多态形态：每个具体硬件 UART 实现一组 `platform_uart_ops_t` 函数，挂到 `platform_uart_dev_t.uart_ops`。SPI、I²C 同结构。

I²C 抽象用 message 数组的形式（前一章 max31827 用过）：

```c
typedef struct {
	uint16_t addr;       /* 设备地址 */
	uint16_t flags;      /* PLATFORM_I2C_WR / PLATFORM_I2C_RD */
	uint16_t len;
	uint8_t *buf;
} platform_i2c_msg_t;

uint32_t platform_i2c_transfer(platform_i2c_bus_device_t *bus,
                               platform_i2c_msg_t *msgs, uint32_t num);
```

一次 `platform_i2c_transfer(bus, msgs, 2)` 完成"先写寄存器地址、再读两字节"的标准 I²C 序列。Linux 内核 `i2c_transfer` 同款 API。

SPI 抽象用 `platform_spi_device_t` 表示一个具体片选下的设备（不是 SPI 总线）：每个 SPI 设备绑定一根 CS 引脚 + 时钟相位 / 极性配置。一条 SPI 总线挂多个设备时，每个设备一个 `platform_spi_device_t` 实例。

应用层一行查表 `platform_device_find("uart3")` 拿到设备指针，剩下全靠 vtable 派发。**换 MCU 不改一行应用层代码**。这是第 15 章"换硬件不改应用"的真实落地，比 ch15 教学版的样子复杂多了，但抽象骨架完全一样。

## 20.7 跨 MCU 移植清单：什么会变 / 什么不会变

子控板现在跑在 STM32H743。如果要移植到 i.MX RT1170（恩智浦 Cortex-M7）或者 GD32H7（兆易创新国产 Cortex-M7），具体改什么？

**一定不会变（占代码量约 85%）：**

- 所有 driver（h_motor / encoder / position_switch / led 等等），它们只调 platform 层接口
- 所有应用层逻辑（状态机、协议解析、和主控通信）
- 所有 OOP 抽象（base / ops / 子类）
- 行业认证流程相关的所有代码（参数校验、错误处理、日志）

**会变（占代码量约 10%）：**

- `platform/arch/` 下的 ARM 体系结构相关代码（中断向量、低功耗、缓存控制），不同厂家虽然同是 Cortex-M7，外设地址映射不同
- `platform/board/` 下的这块板的引脚映射、时钟树配置
- 启动文件（startup.s）、链接脚本（.ld）

**完全重写（占代码量约 5%）：**

- HAL 库（STM32 用 STM32CubeMX 生成，i.MX 用 NXP MCUXpresso，GD32 用 GD32CubeMX），这部分是 MCU 厂家自己提供的，平台层通过函数指针把它们的差异吸收掉

具体清单：

| 文件类别 | STM32H7 → i.MX RT1170 | STM32H7 → GD32H7 |
|---|---|---|
| `app/drivers/*.c/.h` | 不改 | 不改 |
| `app/actor/*.c/.h` | 不改 | 不改 |
| `app/platform/platform_*.h`（接口层） | 不改 | 不改 |
| `app/platform/arch/cortex_m7/cm7_irq.c` 中断映射 | 部分改：i.MX 中断号不同 | 部分改：GD32 中断号不同 |
| `app/platform/arch/cortex_m7/cm7_cache.c` D-Cache | 不改（同是 CMSIS API） | 不改 |
| `app/platform/board/<board>/pin_table.c` | 全改：GPIO 地址映射不同 | 全改：但和 STM32 高度相似 |
| `app/platform/board/<board>/clock_init.c` | 全改：i.MX 时钟树差异大 | 部分改：GD32 时钟树和 STM32 相似 |
| `Drivers/<MCU厂>_HAL/` 整目录 | 换成 NXP MCUXpresso | 换成 GD32 标准外设库 |
| `STM32H7xxXX_FLASH.ld` 链接脚本 | 改地址（i.MX RT1170 有 SDRAM 域） | 改地址（GD32 Flash 大小不同） |
| `Core/Src/main.c` CubeMX 生成 | 用 MCUXpresso 重新生成 | 用 GD32CubeMX 重新生成 |
| `startup_xxx.s` 启动文件 | 换成 i.MX 启动 | 换成 GD32 启动 |

**具体行级别会变的例子。** 比如某个 UART 实现文件 `platform_hw_uart_stm32h7.c` 里：

```c
/* STM32H7 版本 */
huart->Instance = USART3;
huart->Init.BaudRate = 115200;
HAL_UART_Init(huart);
```

迁移到 i.MX RT1170 这一段变成：

```c
/* i.MX RT1170 版本 */
LPUART_GetDefaultConfig(&config);
config.baudRate_Bps = 115200;
LPUART_Init(LPUART3, &config, CLOCK_GetFreq(kCLOCK_OscRc48MDiv2));
```

平台层把这种差异完全吸收。`platform_uart_ops_t` 的 `write` 实现内部用哪家 HAL 函数完全不影响驱动层和应用层。

这就是为什么这本书前 18 章铺垫的所有 OOP 抽象在工业项目里**真的有用**：8 万行代码移植到新 MCU，只动 5%~15%。如果没有这套抽象，每次换 MCU 等于推倒重写。

## 20.8 子控板的 initcall 顺序

子控板用的 initcall 7 级和主控板一样（都是参考 RT-Thread 实现）。但各级里跑的具体内容不同：

| 级别 | 主控板上的代表 | 子控板上的代表 |
|---|---|---|
| 1 BOARD | STM32 时钟树 + GPIO PINMUX | 同上 |
| 2 PREV | EasyLogger init | EasyLogger init |
| 3 DEVICE | platform_uart / i2c / spi 注册 | 多了 platform_pwm（直驱直流电机用） |
| 4 COMPONENT | Letter Shell + 异步事件框架 | 多了 MultiButton 框架（位置开关用） |
| 5 ENV | LED / 温度 / 风扇 cfg | 水平电机 cfg + 垂直电机 cfg + 编码器 cfg + 位置开关 cfg |
| 6 APP | 业务状态机订阅事件 | 运动控制状态机订阅电机回调 |
| 7 READY | 看门狗启动 | 同上 |

这是同一套机制服务两块完全不同的板。骨架不变，每一级的内容按板上有什么外设填进去。

## 20.9 与教学版的差距小结

把第 19 章的 12 件差距延伸到子控板：

| # | 维度 | 教学版 | 工业版子控板 |
|---|---|---|---|
| 13 | ops 表规模 | 3-5 个虚函数 | 24 个虚函数（h_motor）+ 3 个虚函数（v_motor）平行 |
| 14 | 异步抽象 | 直接调函数 | 工作线程 + 消息队列把异步翻译成同步（每个电机两套队列） |
| 15 | 总线统一抽象 | 各驱动直接调 HAL | platform_uart / spi / i²c 三层 ops 表统一吸收 |
| 16 | 多平面继承 | 单一基类 | h_motor 和 v_motor 平行（不强行共基类） |
| 17 | 工程历史 | 一次写完 | 持续演化，每个字段都对应过去的某个 bug |
| 18 | 跨 MCU 移植成本 | 重写 100% | 重写 5%~15% |

加上主控板的 12 件，总共 18 件差距。每一件都不神秘，每一件都对应一个具体的工程问题。

读完这两章你应该能拿到任何一份工业项目的驱动代码，5 分钟看出它的骨架：基类、ops 表、子类层级、回调机制、initcall 注册、Platform 抽象层。剩下的"它为什么这样写"，可以慢慢读，但**架构看一眼就懂**。

## 20.10 驱动模块完整代码索引

跟第 19 章一样，本章不是"抄一份就能 `make && ./demo`"的工程章。子控板代码量比主控板还大，依赖 RTOS、伺服厂家私有命令协议栈、电机厂家私有寄存器映射，没办法把整块板裁出来塞进 PC 模拟。

这一节把本章每个驱动模块的"完整代码"对应到具体来源（仓库里整理后的接口骨架 + 章节正文里贴的实现节选），让你读完后知道每段代码可以在哪里看到原貌。

### 20.10.1 水平电机驱动 h_motor（20.1 / 20.2 / 20.4 节）

完整骨架位于 [`industrial/motor_24vfuncs/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/motor_24vfuncs/)：

| 文件 | 角色 | 章节正文对应节选 |
|---|---|---|
| `h_motor_base.h` | 24 虚方法 ops 表 + 三种回调 + 应用层接口（`h_motor_base_velocity_set` 等封装函数声明） | 20.1 节"24 个虚方法的 ops 表"代码块 + 20.4 节"三种回调"代码块 |
| `h_motor_acm.h` | ACM 伺服电机子类 struct + 构造参数 | 20.2 节"节选自 `app/drivers/h_motor/h_motor_acm.h`"代码块 |

`h_motor_base.c`（封装函数实现，每个函数体内部走 `me->ops->xxx(me, ...)` dispatch）和 `h_motor_acm.c`（24 个虚方法的具体厂家协议实现 + 工作线程 + 消息队列）**没有放进仓库**。原因有两条：

第一，`h_motor_acm.c` 涉及伺服电机厂家的私有寄存器映射（每个寄存器地址、bit 含义、操作时序），这部分是商业敏感信息，整理后再发表的成本高于教学收益。

第二，工作线程 + 消息队列的具体实现绑定在 RTOS 上（CMSIS-RTOS2 API），脱离 RTOS 移植到 PC 模拟需要重写一遍同步原语，对 OOP 教学价值有限。

20.2 节正文已经把 OOP 角度的全部要点讲清楚：

- 子类把 `h_motor_base_t base` 放第一字段（向上转型零代价）
- 工作线程 + 消息队列把异步操作翻译成"看起来同步"的接口
- 故障检测引脚通过 `Button` 库当成"按键事件"统一处理
- 性能监控字段（`max_following_error / max_motor_current / max_i2t_sum`）持续记录峰值
- `i2t_reg_unsupported` 标志位适配同芯片不同固件
- 命令排序号 `last_work_id` 配合调试

20.1 节贴的 `acm_velocity_set / acm_following_err_limit_set / acm_i2t_peak_current_set` 三个函数节选示意了"虚方法 → 工作队列 → 厂家协议组帧 → UART 发送"的标准模式，照这个模式可以补出剩下 21 个虚方法。

### 20.10.2 垂直电机驱动 v_motor（20.3 节）

20.3 节贴的是 `v_motor_base.h` 基类（3 虚方法 ops 表 + 2 回调）+ `v_motor_ifx007t.h` 子类 struct。**完整 `.c` 实现已经放进仓库**[`industrial/motor_24vfuncs/v_motor_ifx007t.c`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/motor_24vfuncs/v_motor_ifx007t.c)，是产品级版本。

20.3 节正文用最小内核描述了行为：

> `motor_move(UP)` 在子类里就是把 `ctrl_1_pin` 拉高、`ctrl_2_pin` 拉低、两个 sleep 拉高使能；`motor_move(DOWN)` 反过来；`motor_move(STOP)` 把两个 ctrl 都拉低。

这是教学版讲清楚 OOP 抽象骨架的最简形态。仓库里的产品级 `v_motor_ifx007t.c` 在这个骨架上多了五件工程细节：

1. PWM 调速代替直接 GPIO 推挽（H 桥两路 PWM）
2. 软启动（5% → 起步占空，避免上电瞬间冲击电流）
3. 满速线性加速（起步占空 → 100%）
4. 减速区限位 + 三层超时保护
5. IS 引脚电流采样 + EMA 滤波 + 连续超阈过流故障判断

OOP 骨架完全不变：`v_motor_base_t` 的 3 个 ops + 1 个错误回调，子类内部把所有这些异步行为塞进工作线程 + 周期定时器。应用层永远只见 `v_motor_base_move(motor, UP)` 这一句普通函数调用。这就是 ch20 想讲透的核心：工程现实里"几行 GPIO" 之上扛住的细节再多，OOP 抽象层一字不动。

### 20.10.3 编码器 / 位置开关（20.5 节）

20.5 节贴的是 `encoder.h` + `position_switch.h` 两个 struct 定义和回调签名。**完整 `.c` 实现没有放进仓库**。

20.5 节正文要传达的核心是反面教材：单实现驱动不要硬上 ops 表（`struct + me` 就够），多个相似硬件信号（5 个限位开关）要打包成一个 struct + enum 区分而不是 5 份独立 driver。这两条 OOP 原则跟具体 `.c` 实现无关，所以仓库里没有放完整实现。

### 20.10.4 Platform 层 UART / I²C / SPI（20.6 节）

UART 抽象骨架位于 [`industrial/platform_layer/platform_uart.h`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/platform_layer/platform_uart.h)：

| 文件 | 角色 | 章节正文对应节选 |
|---|---|---|
| `platform_uart.h` | UART ops 表 + 设备注册接口 | 20.6 节"UART 抽象的接口"代码块 |
| `platform_module_export.h` | initcall 7 级宏（第 19 章已用过） | 第 19 章 19.6 节 |

I²C / SPI 的接口在仓库里没有单独贴出来，因为骨架跟 UART 同构（设备 struct + ops 表 + 字符串名字注册表查找）。20.6 节正文里贴的 `platform_i2c_msg_t` 结构和 `platform_i2c_transfer` 接口是 I²C 抽象的核心；SPI 抽象按 `platform_spi_device_t` 一句话讲过：每个具体片选下的设备一个实例，绑定 CS 引脚 + 时钟相位 / 极性。

具体硬件后端（`platform_uart_stm32h7.c` / `platform_uart_imxrt.c` 等）依赖各家 MCU 的 HAL 库，跟 OOP 教学主线无关，仓库里不放。

### 20.10.5 完整工程参考

如果你想看一份**完整、可跑、跨 STM32 / Linux 双后端**的工程（不是接口骨架而是完整可编译版本），看附录 B 和附录 C：

- 附录 B：`industrial/stm32_full/` STM32F407 4 颗 LED 流水灯，跑通 ch01-ch17 全部 OOP 抽象

### 一句话总结

本章把工业 OOP 抽象能力的天花板（24 虚方法 + 异步翻译同步 + 跨 MCU 移植）讲完了。每个驱动模块对应到 `industrial/motor_24vfuncs/` 和 `industrial/platform_layer/` 下的真实接口骨架 + 章节正文里的实现节选。完整可跑的小工程在附录 B / C 的 `industrial/stm32_full/` 和 `industrial/linux_full/`，要看完整伺服电机驱动的现实形态本章正文 + `industrial/motor_24vfuncs/` 已经齐全。
