# 第 19 章 · 工业控制板主控案例

参考代码：[`industrial/led_basic/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/industrial/led_basic/) 是 LED 驱动的最小骨架，本章其它驱动（温度传感器、EEPROM、风扇）的关键代码段全部内联在正文里。

前面 18 章一颗 LED 一步步演化出来的所有抽象，到这一章你会看到它们在真实工业项目里实际长什么样。

代码来自我参与开发的工业控制板项目。两块板：主控板和子控板，合起来约 80K 行 C 代码。本章讲主控板，下一章讲子控板。

主控板站在系统上层。它跟外面的 Host 主机通过 TCP/IP 通信，跟内部的子控板通过串口通信，自己直接管的外设包括 LED、EEPROM、风扇、激光、蜂鸣器、温度传感器、湿度传感器、屏蔽门检测、电压采样等十几路。本章挑四个最能说明问题的驱动来看：LED（最小继承范例）、EEPROM（多回调与 POSIX 风格返回值）、温度传感器（两层继承 + 多种芯片类型）、风扇（无 ops 表的 struct + me 形态）。

为了让你从"驱动"看到"整块板的工程组织"，本章后半段还会展开三件工业级机制：设备如何被全局命名（设备句柄注册表）、驱动如何自动按依赖顺序启动（initcall 7 级）、主控板用了哪些中间件（环形缓冲、命令行 shell、日志系统）。这三件事是真实工业代码和教学版之间最具体的差距。

## 19.1 LED 驱动：最小继承范例

挂号单一路推到工业代码会怎样？看这一段（节选自工业项目 `app/drivers/led/led_base.h`）：

```c
struct led_base;

struct led_ops {
	void (*led_on)(struct led_base *me);
	void (*led_off)(struct led_base *me);
};

struct led_base {
	const struct led_ops *ops;
};

void led_on(struct led_base *me);
void led_off(struct led_base *me);
```

第 11 章你推出来的 `struct led_base + struct led_ops + ops 字段` 三件套，工业代码里一字不差。整个仓库统一用 Linux 内核风格 `struct led_base`，不上 typedef 别名。第 1 章 1.7.3 节解释过为什么。

`struct led_base` 是基类。它只放公共字段：一个指向 ops 表的指针。任何具体的 LED（GPIO 拉线的、PWM 调亮度的、I²C 寄存器写的）都可以子类化它。

`led_base.c` 实现就两行：

```c
void led_on(struct led_base *me)
{
	return me->ops->led_on(me);
}

void led_off(struct led_base *me)
{
	return me->ops->led_off(me);
}
```

这是第 11 章的多态完整图景。`me->ops->led_on(me)` 一行，背后是 vtable 查表加间接跳转。

GPIO 类型的子类（节选自工业项目 `app/drivers/led/led_gpio.h`）：

```c
struct led_gpio {
	struct led_base base;        /* 基类放在第一个字段 */
	int32_t         pin_num;
	bool            light_level; /* 高电平点亮还是低电平点亮 */
};

platform_err_t led_gpio_init(struct led_gpio *me,
                             const char *pin_name,
                             bool light_level);
```

子类实现（节选自 `led_gpio.c`）：

```c
static void led_gpio_on(struct led_base *me)
{
	struct led_gpio *led_gpio = (struct led_gpio *)me;
	platform_pin_write(led_gpio->pin_num, led_gpio->light_level);
}

static void led_gpio_off(struct led_base *me)
{
	struct led_gpio *led_gpio = (struct led_gpio *)me;
	platform_pin_write(led_gpio->pin_num, !led_gpio->light_level);
}

static const struct led_ops ops = {
	.led_on  = led_gpio_on,
	.led_off = led_gpio_off,
};

platform_err_t led_gpio_init(struct led_gpio *me,
                             const char *pin_name,
                             bool light_level)
{
	if (!me || !pin_name)
		return PLATFORM_EINVAL;

	int32_t pin_num = platform_pin_get(pin_name);
	if (pin_num < 0)
		return PLATFORM_EINVAL;

	platform_pin_mode(pin_num, PIN_MODE_OUTPUT);
	platform_pin_write(pin_num, !light_level);

	me->pin_num     = pin_num;
	me->light_level = light_level;
	me->base.ops    = &ops;
	return PLATFORM_EOK;
}
```

四个细节值得注意。

**一，基类放在第一个字段。** `struct led_gpio` 的第一个字段是 `struct led_base base`。这样 `(struct led_base *)led_gpio` 强转是合法的（地址相同），向上转型代价为零。这是第 12 章你已经推过的事。

**二，子类向下转型用 `(struct led_gpio *)me`。** 因为 `base` 在第一个字段，子类指针和基类指针的数值地址相同。如果 `base` 不在第一个字段，就必须用 `container_of`。第 13 章讲过。

**三，ops 表是 `static const`。** `static const struct led_ops ops = { ... }` 全局唯一一份，所有 `struct led_gpio` 实例共享。这就是 C++ 编译器为每个 class 偷偷生成的 vtable，工业代码里手动写出来。

**四，`light_level` 这个字段是工程实践才会有的。** 教学版只想"点亮 LED"，工业版要面对一个事实：硬件设计上，有的 LED 是高电平点亮（push-pull），有的是低电平点亮（共阳极接法）。`light_level = true` 表示高电平点亮。同一份 `led_gpio_on` 函数，靠这个字段服务两种硬件设计。

**配置代码长什么样。** `struct led_gpio` 实例不是在 `main` 里定义的，是在专门的配置文件里。节选自 `app/environment_cfg/led_cfg.c`：

```c
#include "platform/platform_module_export.h"
#include "led/led_gpio.h"

struct led_base *green_led = NULL;
struct led_base *red_led   = NULL;

static struct led_gpio green_led_instance;
static struct led_gpio red_led_instance;

static void led_cfg(void)
{
	platform_err_t ret;

	ret = led_gpio_init(&green_led_instance, "PI.12", 0);
	if (ret != PLATFORM_EOK) {
		log_e("green led init failed");
		return;
	}
	green_led = (struct led_base *)&green_led_instance;

	ret = led_gpio_init(&red_led_instance, "PI.14", 0);
	if (ret != PLATFORM_EOK) {
		log_e("red led init failed");
		return;
	}
	red_led = (struct led_base *)&red_led_instance;
}

INIT_ENV_EXPORT(led_cfg);
```

应用层调用（在 `environment_cfg/environment_export.h` 里 extern 句柄之后）：

```c
extern struct led_base *green_led;
extern struct led_base *red_led;

led_on(green_led);
led_off(red_led);
```

`green_led` 和 `red_led` 都是 `struct led_base *` 类型。应用层不需要知道它们是 GPIO 类型还是 PWM 类型还是 I²C 类型。这是第 12 章的向上转型威力：一个基类指针指所有 LED。

`INIT_ENV_EXPORT(led_cfg)` 是 19.6 节会展开的 initcall 注册宏，这里你先记一句话：链接器会把所有 `INIT_ENV_EXPORT` 的函数收集到一个段里，系统启动时自动调用。`main()` 不需要写一行 `led_gpio_init`。

教学版（`oop-in-c/code/01-three-leds/`）和工业版的对比：

| 维度 | 教学版 ch01 | 工业版 |
|---|---|---|
| 基类 | `struct led { uint8_t pin; ... }` | `struct led_base { ops* }` |
| 子类 | 没有 | `struct led_gpio { base, pin_num, light_level }` |
| 接口 | 直接调用 `led_on(&red_led)` | `led_on(red_led)` 内部 vtable 派发 |
| 拉线方向 | 假设高电平点亮 | 字段 `light_level` 运行时区分 |
| 引脚号类型 | `uint8_t` | 字符串 `"PI.15"` 经 `platform_pin_get` 查表 |
| 实例定义 | `main()` 里手动 `struct led red;` | `*_cfg.c` 文件里 `static struct led_gpio red_instance;` |
| 启动顺序 | `main()` 里手动调 init | `INIT_ENV_EXPORT(led_cfg)` 自动收集 |

教学版到这里就停了。工业版还要继续走：通过 ops 表，可以把 PWM 调亮度的 LED 子类、I²C 寄存器写的 LED 子类挂到同一个基类下，应用层一行不改。

## 19.2 EEPROM：POSIX 风格的随机访问接口

LED 简单。它没有"事件"，只有"操作"。但凡涉及外设有自主行为（写入完成、数据上来了、出错了），就需要回调。

EEPROM 是介于两者之间的过渡例子（节选自 `app/drivers/eeprom/eeprom_base.h`）：

```c
struct eeprom_base;

struct eeprom_ops {
	int32_t (*eeprom_read)(struct eeprom_base *dev, uint32_t pos,
	                       uint8_t *buffer, uint32_t size);
	int32_t (*eeprom_write)(struct eeprom_base *dev, uint32_t pos,
	                        const uint8_t *buffer, uint32_t size);
};

struct eeprom_base {
	const struct eeprom_ops *ops;
};

int32_t eeprom_read(struct eeprom_base *dev, uint32_t pos,
                    uint8_t *buffer, uint32_t size);
int32_t eeprom_write(struct eeprom_base *dev, uint32_t pos,
                     const uint8_t *buffer, uint32_t size);
```

骨架和 LED 一样：基类只放 ops，所有具体芯片（DS2433 1-Wire EEPROM、I²C EEPROM、SPI EEPROM）都子类化 `struct eeprom_base`。区别在 ops 表里多了 `pos` 和 `size` 参数：随机访问、可读可写。

为什么 read/write 返回 `int32_t` 不是 `bool`？返回值约定如下：

| 返回值 | 含义 |
|---|---|
| >= 0 | 实际读 / 写的字节数 |
| < 0 | 错误码（`PLATFORM_EINVAL` 等） |

这个约定借鉴 POSIX `read(2)` / `write(2)` 系统调用。教学版会写 `bool ok = eeprom_write(...)`，工业版必须区分"写了多少"，原因有三个。

第一，写跨页的情况。DS2433 这颗 EEPROM 一页 32 字节。应用层想从地址 30 写 50 字节，物理上跨两页，需要拆成两次硬件写。如果中间一次失败，已经写了多少必须告诉上层，让上层决定回滚还是重试。

第二，写到容量边界。DS2433 总容量 4096 比特（512 字节）。从地址 500 写 50 字节，到 511 就到底了。底层只能写 12 字节，剩下 38 字节扔掉。返回 12 让上层知道实际成功的字节数。

第三，写比读慢。EEPROM 写一次要几毫秒（比读慢 100 倍）。底层支持"超时返回"，写到一半超时就返回已写入的字节数。教学版的 bool 接口没法表达这种部分成功。

`int32_t` 的负数空间留给错误码。`PLATFORM_EINVAL = -1`、`PLATFORM_EIO = -2`、`PLATFORM_ETIMEOUT = -3` 等。应用层模式：

```c
int32_t n = eeprom_read(sn_eeprom, 0, buf, 32);
if (n < 0) {
	log_e("eeprom read failed: %d", n);
	return -1;
}
if ((uint32_t)n < 32) {
	log_w("eeprom read short: %d of 32", n);
}
/* 用前 n 字节 */
```

应用层句柄声明（`environment_cfg/environment_export.h`）：

```c
extern struct eeprom_base *sn_eeprom;
```

`sn_eeprom` 用来存板级序列号。底层接的是 DS2433（1-Wire EEPROM 芯片），但应用层不知道也不需要知道。如果将来某一版硬件换成 I²C EEPROM，只要再写一个子类 `struct eeprom_i2c`，让 `sn_eeprom` 指过去，应用层一行不动。这是第 15 章"换硬件不改应用"在工业项目里的具体落地。

## 19.3 温度传感器：两层继承 + 同基类下挂多种芯片

LED 的子类只有一种（`struct led_gpio`）。EEPROM 的子类有两种（DS2433、I²C），但项目里只用了一种实现，没真正体现"多个子类共用基类"的威力。温度传感器是主控板上第一个真正多子类的驱动。

**第一层：基类** 节选自 `app/drivers/temp_sensor/temp_sensor_base.h`：

```c
struct temp_sensor_base;

struct temp_sensor_ops {
	platform_err_t (*temp_read)(struct temp_sensor_base *dev,
	                            float *temp_out);
};

struct temp_sensor_base {
	const struct temp_sensor_ops *ops;
};

platform_err_t temp_sensor_read(struct temp_sensor_base *dev,
                                float *temp_out);
```

基类极简：一个 `ops` 字段，一个 `temp_read` 虚方法。这反映温度传感器抽象层面的本质：**给一个度数出来，单位摄氏度**。所有温度传感器芯片，不管 I²C、SPI、单总线，本质都是这一件事。

基类实现（节选自 `temp_sensor_base.c`）：

```c
platform_err_t temp_sensor_read(struct temp_sensor_base *dev, float *temp_out)
{
	platform_err_t ret = PLATFORM_EOK;

	if ((NULL == dev) || (NULL == temp_out)) {
		ret = PLATFORM_EINVAL;
		goto exit;
	}

	if ((NULL == dev->ops) || (NULL == dev->ops->temp_read)) {
		ret = PLATFORM_ENOSYS;
		goto exit;
	}

	ret = dev->ops->temp_read(dev, temp_out);

exit:
	return ret;
}
```

注意基类做了两层校验：先校验 `dev` / `temp_out` 是不是 `NULL`，再校验 `ops` 和 `ops->temp_read` 是不是 `NULL`。教学版只写第一层 `if (!me) return -1;`，工业版多写第二层是因为子类构造函数可能没赋 ops（编程错误），多走一层校验在出错时给出明确的 `PLATFORM_ENOSYS`（"没这个虚方法"），调试时一眼定位。

**第二层：max31827 子类** 节选自 `app/drivers/temp_sensor/max31827.h`：

```c
#include "temp_sensor_base.h"
#include "platform/platform_i2c.h"

struct max31827 {
	struct temp_sensor_base base;
	uint8_t dev_addr;
	struct platform_i2c_bus_device *i2c_bus;
};

platform_err_t max31827_init(struct max31827 *dev,
                             const char *i2c_bus_name,
                             uint8_t dev_addr);
```

MAX31827 是 Analog Devices（原 Maxim）一颗 I²C 接口的数字温度传感器，12 位分辨率（LSB 0.0625 度），量程 -40 度到 +125 度。子类只多两个字段：I²C 总线句柄 + 设备地址。这就是"具体芯片要的额外信息"。

子类实现（节选自 `max31827.c`）：

```c
#define TEMP_RESOLUTION_FOR_12_BIT  (0.0625f)
#define TEMPERATURE_MASK            (0x0FFF)

static platform_err_t
max31827_temp_read(struct temp_sensor_base *dev, float *temp_out);

static const struct temp_sensor_ops max31827_ops = {
	.temp_read = max31827_temp_read,
};

platform_err_t
max31827_init(struct max31827 *dev, const char *i2c_bus_name,
              uint8_t dev_addr)
{
	if ((NULL == dev) || (NULL == i2c_bus_name))
		return PLATFORM_EINVAL;

	dev->i2c_bus = platform_i2c_bus_device_find(i2c_bus_name);
	if (NULL == dev->i2c_bus)
		return PLATFORM_EIO;

	dev->dev_addr = dev_addr;
	dev->base.ops = &max31827_ops;

	return PLATFORM_EOK;
}

static platform_err_t
max31827_temp_read(struct temp_sensor_base *dev, float *temp_out)
{
	uint8_t tx_buf[1] = {0x00};       /* 寄存器地址：温度寄存器 */
	uint8_t rx_buf[2] = {0x00, 0x00};
	struct platform_i2c_msg msg[2];
	float temp;

	struct max31827 *self = (struct max31827 *)dev;  /* 向下转型 */

	msg[0].addr  = self->dev_addr;
	msg[0].buf   = tx_buf;
	msg[0].len   = 1;
	msg[0].flags = PLATFORM_I2C_WR;

	msg[1].addr  = self->dev_addr;
	msg[1].buf   = rx_buf;
	msg[1].len   = 2;
	msg[1].flags = PLATFORM_I2C_RD;

	if (platform_i2c_transfer(self->i2c_bus, msg, 2) != 2)
		return PLATFORM_EIO;

	uint16_t raw_data = (rx_buf[0] << 8) | rx_buf[1];

	/* 第 16 位是符号位，负温度走二的补码 */
	if (raw_data & (1 << 15)) {
		raw_data = (raw_data ^ 0xFFFF) + 1;
		temp = -(float)raw_data * TEMP_RESOLUTION_FOR_12_BIT;
	} else {
		raw_data &= TEMPERATURE_MASK;
		temp = (float)raw_data * TEMP_RESOLUTION_FOR_12_BIT;
	}

	*temp_out = temp;
	return PLATFORM_EOK;
}
```

这一份子类完整体现了所有教学版没讲透的工程细节，逐个看。

**一，向下转型在第一行就发生。** `struct max31827 *self = (struct max31827 *)dev;` 是 ch12-ch13 讲过的"基类指针拿回子类指针"。因为 `struct temp_sensor_base base` 是 `struct max31827` 的第一个字段，地址相同，强转合法。如果不放第一个字段，就必须用 `container_of`。

**二，I²C 协议被压成两条 message。** I²C 读外设寄存器的标准流程是"先写寄存器地址、再读 N 字节"。把这两步打包成 `platform_i2c_msg_t` 数组传给 `platform_i2c_transfer`，底层一次性发出，省去用户层处理 START / STOP / RESTART 的细节。返回值是"成功传输的 message 数"，等于 2 就成功。

**三，原始数据是 16 位定点数。** MAX31827 的温度寄存器格式：高字节先来，低字节后来，最高位是符号位，剩下 15 位中只有低 12 位有效（分辨率 0.0625 = 1/16，意味着最低 4 位表示小数部分，但寄存器格式把它编码到高位，所以读出来直接乘 0.0625 就得到摄氏度）。负数走二的补码：取反加一，再加上负号。

**四，子类的 ops 是 `static const`。** `static const struct temp_sensor_ops max31827_ops = { .temp_read = max31827_temp_read };` 全局唯一一份。即便系统里挂了 8 颗 MAX31827，也共用这一份 ops 表。这是 vtable 在工业代码里的标准写法。

**配置代码** 节选自 `app/environment_cfg/temp_sensor_cfg.c`：

```c
#include "platform/platform_module_export.h"
#include "temp_sensor/max31827.h"

struct temp_sensor_base *temp_board = NULL;

static void temperature_sensor_cfg(void)
{
	static struct max31827 max31827_dev;

	platform_err_t ret = max31827_init(&max31827_dev, "i2c4", 0x5A);
	if (ret != PLATFORM_EOK) {
		log_e("max31827 init failed");
		return;
	}

	temp_board = (struct temp_sensor_base *)&max31827_dev;
}

INIT_ENV_EXPORT(temperature_sensor_cfg);
```

`temp_board` 在 `environment_export.h` 里 extern 出来，应用层用：

```c
float t = 0.0f;
if (temp_sensor_read(temp_board, &t) == PLATFORM_EOK) {
	if (t > 70.0f)
		log_w("board overheating: %.2f", t);
}
```

应用层只调基类接口 `temp_sensor_read`。它不知道底下接的是 MAX31827，也不知道走的是 I²C bus 4。

**为什么不直接两层都合并到一层？** 这个问题最容易问。"反正现在只有一颗 MAX31827，我直接写 `max31827_read(dev, &t)` 不就好了，何必套个基类？"

答案在硬件演化。同一块板第二版可能换 SHT31（带温湿度的 I²C 传感器），第三版可能换 NCT75（更便宜的 I²C 传感器），第四版可能为了高精度场合换 MAX31875（19 位精度）。每换一颗芯片，应用层（"过 70 度报警"那段逻辑）一行不动。新增子类挂上去，cfg 文件里 `temp_board` 指向新子类即可。

教学版常见的错是看到只有一颗芯片就不抽基类，第二颗芯片来的时候发现应用层每个用到温度的地方都要改一遍，最后变成复制粘贴。基类抽象的成本是"多一层指针跳转"，收益是"任何时候硬件方案换芯片不改应用层"。工业项目里这笔账 99% 是值得做的。

**和教学版 `struct led` 的差距详细对比。** 把这一节的 `struct max31827` 和 ch01 的 `struct led` 摆一起，能数出教学版到工业版具体多了什么：

| # | 维度 | 教学版 `struct led` | 工业版 `struct max31827` |
|---|---|---|---|
| 1 | 是否有继承层级 | 没有 | 两层（base + max31827） |
| 2 | 是否有虚函数表 | 没有 | 有 `struct temp_sensor_ops` |
| 3 | 字段数 | 3（pin/brightness/is_on） | 3（base/dev_addr/i2c_bus）但语义层级不同 |
| 4 | 接口暴露范围 | 一颗具体 LED | 整类温度传感器 |
| 5 | 错误处理 | `return -1` | `platform_err_t` 共 10+ 种错误码 |
| 6 | 总线抽象 | 直接 `platform_pin_write` | 经 `platform_i2c_bus_device_find` 查表后 `platform_i2c_transfer` |
| 7 | 设备地址 | 引脚号 `uint8_t` | I²C 地址 + bus 句柄 |
| 8 | 数据解码 | 无（写 1 / 0） | 16 位定点数解析 + 符号位 + 二的补码 |
| 9 | 实例创建 | `struct led red;` 全局 | `static struct max31827 dev;` 在 cfg 文件里 |
| 10 | 句柄暴露 | 直接 `extern struct led red;` | `extern struct temp_sensor_base *temp_board;` |
| 11 | 启动注册 | `main()` 里调 init | `INIT_ENV_EXPORT(cfg_func)` 自动 |

这 11 件事单独看每一件都不复杂。挤在一起就是教学到工业的全部差距。

## 19.4 风扇驱动：状态机与状态变化回调

风扇控制比 LED 复杂在它有状态机。三档转速（低、中、高）加上停转状态，加上风扇本身可能因为外部原因失速。风扇驱动需要把状态变化主动告诉上层。

节选自 `app/drivers/fan/fan.h`：

```c
struct fan;

typedef void (*fan_status_change_cb_t)(struct fan *me, uint32_t status);

struct fan {
	platform_device_t      uart_dev;
	fan_status_change_cb_t status_change_cb;
	uint32_t               status;
};

void fan_init(struct fan *me, const char *uart_name);
platform_err_t fan_ctrl(struct fan *me, uint32_t level);
uint32_t fan_status_get(struct fan *me);
void fan_status_change_cb_register(struct fan *me,
                                   fan_status_change_cb_t cb);
```

注意三件事。

**一，没有 ops 表。** 这个项目里风扇只接一种型号（UART 串口控制的智能风扇），没有第二种实现，所以没必要套虚函数表。OOP 不是规则，是工具。能简单解决的问题就别上多态。教学版常见的错误是"把所有 driver 都套上 ops 表"，工业代码不会这么干。这条规则反过来说也成立：将来真的多了一种风扇（比如 PWM 直驱的简单风扇），再把这个 `struct fan` 升级成 `struct fan_base` + 子类，迁移成本可控。

**二，仍然是 struct + me。** `struct fan` 有自己的字段（uart 设备句柄、回调指针、当前状态），所有函数第一个参数都是 `struct fan *me`。这就是第 1 章的封装基本盘，没有 ops 表也是封装。

**三，状态变化回调。** `fan_status_change_cb_register` 让上层注册一个函数：风扇档位变化时被调用。教学版思路是 `if (last_status != cur_status) printf(...)` 写在风扇内部，工业版是"我变了我告诉你，怎么处理是你的事"。这是依赖反转的最朴素形态。风扇模块不依赖具体的上层（应用层、状态机、日志系统），它只负责把变化事件喷出来。

应用层注册：

```c
static void on_fan_status_changed(struct fan *me, uint32_t status)
{
	(void)me;
	switch (status) {
	case FAN_STATUS_OFF:    log_i("fan stopped"); break;
	case FAN_STATUS_LOW:    log_i("fan low");     break;
	case FAN_STATUS_MID:    log_i("fan mid");     break;
	case FAN_STATUS_HIGH:   log_i("fan high");    break;
	case FAN_STATUS_FAULT:  log_e("fan fault!");  break;
	default: break;
	}
}

fan_status_change_cb_register(fan_dev, on_fan_status_changed);
```

风扇驱动内部的事件喷射逻辑（节选）：

```c
static void fan_status_update_internal(struct fan *me, uint32_t new_status)
{
	if (me->status == new_status)
		return;     /* 没变化不喷 */

	me->status = new_status;

	if (me->status_change_cb)
		me->status_change_cb(me, new_status);
}
```

`if (me->status_change_cb)` 这一行的工程意义比看起来重要：上层没注册回调就不调用，避免空指针解引用。教学版常忘这一行，是工业代码 NULL 检查里最容易漏的。

## 19.5 设备句柄注册：environment_export.h 这一份"全局名片夹"

驱动有了，应用层怎么找到具体的实例？教学版通常在 `main.c` 里 `struct led red_led;` 直接定义全局变量。工业项目几十个驱动几百个实例，全堆在 `main.c` 行不通。

主控板项目用的方法是：每个驱动实例在自己的 `*_cfg.c` 文件里定义、在 `environment_cfg/environment_export.h` 里 extern 出来。节选：

```c
/* environment_cfg/environment_export.h */
extern struct led_base *green_led;
extern struct led_base *red_led;

extern struct eeprom_base    *sn_eeprom;
extern struct fan            *fan_dev;
extern struct laser_base     *laser;
extern struct temp_sensor_base *temp_board;
```

应用层写：

```c
#include "environment_cfg/environment_export.h"

led_on(green_led);
fan_ctrl(fan_dev, 2);
temp_sensor_read(temp_board, &t);
```

不需要知道这些设备在哪里被构造、用什么参数构造。`environment_export.h` 是一份"全局名片夹"，所有跨模块共享的对象都在这里登记一次。

注意类型：`struct led_base *`、`struct eeprom_base *`、`struct temp_sensor_base *`，全部是基类指针。应用层只能调基类接口（`led_on`、`eeprom_read`、`eeprom_write`、`temp_sensor_read`），不能调子类特有的方法。这是面向接口编程。

### 19.5.1 这一套机制是 Linux 设备树的精神同款

读到这里你可能觉得这个 `*_cfg.c` 加 `extern struct *_base *handle;` 的写法很眼熟。它跟 Linux 设备树（Device Tree）和 Zephyr 的 `DEVICE_DT_INST_DEFINE` 是同一个解决思路的不同实现。

Linux 设备树是一个 `.dts` 文本文件（编译成二进制 `.dtb`），描述系统里有哪些设备、各自挂哪条总线、什么参数。比如树莓派的某段 dts：

```
i2c1: i2c@7e804000 {
    compatible = "brcm,bcm2835-i2c";
    status     = "okay";
    clock-frequency = <100000>;

    temp@5a {
        compatible = "maxim,max31827";
        reg = <0x5a>;
    };
};
```

启动时内核解析 dtb，看到一颗"挂在 i2c1 上、地址 0x5a、型号 max31827 的温度传感器"，就调 max31827 驱动的 probe 函数把它实例化出来。

Zephyr 类似，但是 dts 在编译期被预处理器宏展开（`DEVICE_DT_INST_DEFINE`），运行时不再有解析过程。

主控板项目没用设备树（项目当年用的 RTOS 不支持），但用的是另一种方法表达同样的东西：

| 维度 | Linux 设备树 | Zephyr DT 宏 | 主控板项目 |
|---|---|---|---|
| 描述形式 | `.dts` 文本文件 | `.dts` 编译期展开 | `*_cfg.c` C 文件 |
| 解析时机 | 启动时运行时解析 dtb | 编译期 | 启动时调 cfg 函数 |
| 描述内容 | compatible / reg / 总线 | 同左 | bus name 字符串 + 设备地址 |
| 句柄获取 | `device_get_binding(...)` | `DEVICE_DT_GET(...)` | `extern struct *_base *xxx;` |
| 加新设备 | 改 dts 重新编译 | 改 dts 重新编译 | 加 `*_cfg.c` 文件 |

三种方案的本质是同一件事：**把"系统里有哪些设备 + 各自参数"从代码里抽出来，放到一份独立的描述文件里**。这样换板子（甚至换芯片）只动这份描述文件，应用层和驱动层都不动。

主控板项目的 `*_cfg.c` 比 Linux 设备树多了一点灵活性：它就是 C 代码，可以放任何初始化逻辑（比如根据 ADC 的某个引脚电平决定挂 EEPROM 子类 A 还是子类 B）。少了一点：纯文本 dts 可以不重新编译就改硬件配置（特别是 Linux 系统），`*_cfg.c` 改了就要重新编译固件。两边都对，是不同语境（Linux 桌面 / 服务器 vs 嵌入式 RTOS 单板）下的取舍。

读完这一节，你拿起任何一份 Linux 内核驱动代码，看到 `of_match_table` 和 `compatible` 字符串，心里就清楚：这是工业项目 `*_cfg.c` 文件的内核版变体。骨架完全一样。

### 19.5.2 platform_device_register / find：底层注册表机制

`green_led` 这个变量是怎么"全局可见"的？答案是：定义在 `led_cfg.c` 里、extern 在 `environment_export.h` 里、应用层 include 后 link 时由链接器解析。这是普通的 C 全局变量。

但有一类设备走更通用的注册表机制：UART、I²C、SPI、ADC 这些总线设备。它们不在 cfg 文件里直接 `extern`，而是用一份运行时注册表。

节选自 `app/platform/platform_device.h`：

```c
struct platform_device {
	const char* name;                            /* "uart3" / "i2c4" */
	uint32_t dev_id;
	uint8_t  ref_count;
	uint16_t flag;
	uint16_t open_flag;
	platform_sys_mutex_t dev_mutex;

	platform_err_t (*rx_indicate)(platform_device_t dev, uint32_t size);
	platform_err_t (*tx_complete)(platform_device_t dev, void *buffer);

	const struct platform_device_ops *ops;
	void *user_data;
};

extern platform_err_t
platform_device_register(platform_device_t dev, const char *name,
                         uint16_t flags);

extern platform_device_t
platform_device_find(const char *name);
```

每个 UART 实例在 board 初始化时调一次：

```c
platform_device_register((platform_device_t)&uart3_dev, "uart3", FLAG);
```

底下 `platform_device_register` 的实现是把这个设备挂到一份全局链表 / 数组里。应用层（或者驱动层）找设备：

```c
platform_device_t uart3 = platform_device_find("uart3");
if (uart3) {
	uart_open(uart3);
	uart_write(uart3, 0, data, len);
}
```

`uart_open` / `uart_write` 是 `platform_uart.c` 里的封装函数（和 1.10 节的 `led_on` 一样），内部走 `uart->ops->open(uart)` / `uart->ops->write(uart, ...)` dispatch 到具体 UART 控制器实现。应用层看到的就是普通函数调用。

这就是 Linux 内核 `bus_register` / `device_add` / `bus_find_device` 的精简版。设备名字（字符串）作为唯一 ID，运行时按名字查找。

为什么 LED 不走这个机制，UART 要走？因为 LED 的句柄是"应用层早就知道哪几个 LED"（设计阶段定的），而 UART 的句柄是"驱动层要按名字找具体硬件资源"（运行期解耦）。前者用 extern 全局变量更直接，后者用注册表更灵活。两种机制并存于工业项目里，按场景选。

这是教学版到工业版第 12 件具体差距。

## 19.6 initcall：让所有驱动自动按顺序启动

`green_led` 这个变量哪里被构造？什么时候被构造？这是工业项目特别讲究的事。

教学版的做法是 `main()` 里手动调一长串 init：

```c
int main(void)
{
	platform_init();
	led_gpio_init(&green_led_obj, "PI.15", true);
	led_gpio_init(&red_led_obj, "PI.14", true);
	eeprom_ds2433_init(&sn_eeprom_obj, ...);
	fan_init(&fan_obj, "uart3");
	max31827_init(&temp_sensor_obj, "i2c4", 0x5A);
	/* ... 几十行 ... */
	app_main();
}
```

随着驱动增多，这个 main 越来越长，每加一个驱动 main 改一次。更糟糕的是依赖关系会乱：A 依赖 B，调换顺序就崩。

主控板项目用的是分级自动初始化机制（参考 RT-Thread 的 INIT_xxx_EXPORT 宏）。节选自 `app/platform/platform_module_export.h`：

```c
typedef struct {
	void (*func)(void);
} module_export_t;

#define INIT_EXPORT(_func, level) \
	__attribute__((used)) const module_export_t \
	module_int##_func \
	__attribute__((section("moduleExport" level))) = \
	{ .func = _func, }

#define INIT_BOARD_EXPORT(fn)         INIT_EXPORT(fn, "1")
#define INIT_PREV_EXPORT(fn)          INIT_EXPORT(fn, "2")
#define INIT_DEVICE_EXPORT(fn)        INIT_EXPORT(fn, "3")
#define INIT_COMPONENT_EXPORT(fn)     INIT_EXPORT(fn, "4")
#define INIT_ENV_EXPORT(fn)           INIT_EXPORT(fn, "5")
#define INIT_APP_EXPORT(fn)           INIT_EXPORT(fn, "6")
#define INIT_SYSTEM_READY_EXPORT(fn)  INIT_EXPORT(fn, "7")
```

每个驱动在自己的 cfg 文件里写一个 init 函数，挂上对应级别的 EXPORT 宏：

```c
/* environment_cfg/led_cfg.c (示意，简化) */
static struct led_gpio green_led_obj;
struct led_base *green_led = (struct led_base *)&green_led_obj;

static void green_led_setup(void)
{
	led_gpio_init(&green_led_obj, "PI.15", true);
}
INIT_DEVICE_EXPORT(green_led_setup);
```

这个 `INIT_DEVICE_EXPORT(green_led_setup)` 展开后会在 ELF 文件里产生一个特殊段 `moduleExport3`，里面只有一个元素：指向 `green_led_setup` 的函数指针。

链接器（linker）把所有 `moduleExport3` 段合并成一段连续的内存区域，放在固件镜像里。系统启动时，遍历这个区域，依次调用每个函数指针：

```c
/* 节选自 app/platform/platform_module_export.c (GCC 工具链版本) */
extern const unsigned int moduleExport3_start;
extern const unsigned int moduleExport3_end;

#define MODULE_EXPORT_EXEC(section)                                  \
	do {                                                             \
		module_export_t *base = (module_export_t *)                  \
			(&moduleExport##section##_start);                        \
		uint32_t cnt = ((unsigned int)(&moduleExport##section##_end) \
		              - (unsigned int)(&moduleExport##section##_start)) \
		              / sizeof(module_export_t);                     \
		for (uint32_t i = 0; i < cnt; i++)                           \
			base[i].func();                                          \
	} while (0)

void platform_module_export_exec(void)
{
	MODULE_EXPORT_EXEC(1);   /* INIT_BOARD_EXPORT */
	MODULE_EXPORT_EXEC(2);   /* INIT_PREV_EXPORT */
	MODULE_EXPORT_EXEC(3);   /* INIT_DEVICE_EXPORT */
	MODULE_EXPORT_EXEC(4);   /* INIT_COMPONENT_EXPORT */
	MODULE_EXPORT_EXEC(5);   /* INIT_ENV_EXPORT */
	MODULE_EXPORT_EXEC(6);   /* INIT_APP_EXPORT */
	MODULE_EXPORT_EXEC(7);   /* INIT_SYSTEM_READY_EXPORT */
}
```

这就是 Linux 内核 4000 万行代码里所有驱动自动启动的同款机制（第 17 章详细讲过）。`__moduleExport3_start` / `_end` 这两个符号不是程序员写的，是链接脚本（`.ld` 文件或 `.sct`）告诉链接器"在 `moduleExport3` 段开头放一个 `start`、结尾放一个 `end`"，运行时就能把整段区域 walk 一遍。

主控板的 7 级层次大致对应：

| 级别 | 名字 | 干什么 | 主控板上的例子 |
|---|---|---|---|
| 1 | INIT_BOARD_EXPORT | 板级初始化（时钟、Flash、外设引脚复用） | STM32 时钟树配置、PINMUX 表 |
| 2 | INIT_PREV_EXPORT | 纯软件预初始化（内存池、日志系统） | EasyLogger init、内存池 init |
| 3 | INIT_DEVICE_EXPORT | 设备驱动初始化（UART / I²C / SPI 总线） | platform_uart_register("uart3"...) |
| 4 | INIT_COMPONENT_EXPORT | 组件初始化（状态机框架、Shell、协议栈） | Shell 初始化、协议栈初始化 |
| 5 | INIT_ENV_EXPORT | 环境配置（具体外设实例：LED、EEPROM、温度传感器） | led_cfg、temp_sensor_cfg、fan_cfg |
| 6 | INIT_APP_EXPORT | 应用层模块（业务逻辑订阅事件、注册回调） | 报警阈值订阅、Host 通信注册 |
| 7 | INIT_SYSTEM_READY_EXPORT | 系统就绪检查 | 看门狗启动、上电完成提示 |

注意区分第 3 级和第 5 级。第 3 级注册的是"总线层级"的设备（UART、I²C、SPI），第 5 级注册的是"挂在总线上的具体外设"（LED、温度传感器）。具体外设依赖总线先 ready，所以必须排在后面。

为什么必须分级？因为依赖。LED 驱动依赖 platform_pin（板级），所以 LED 必须在 BOARD 之后。温度传感器依赖 I²C 总线（设备级），所以 ENV 必须在 DEVICE 之后。状态机订阅传感器事件，所以 APP 必须在 ENV 之后。

加新驱动只要写一行 `INIT_ENV_EXPORT(my_new_driver_setup)`。**不动** main，**不动**任何已有文件。这是开闭原则在 C 工程里最干净的落地。

### 19.6.1 initcall 演进的三个阶段

主控板用到的这套 7 级分级注册机制，不是凭空出现的。它是 C 工程界几十年逐步演化出来的形态。教学版到工业版之间，工程师走过的路一般是这三步：

**阶段一：main 里手动堆。** 项目刚起步、只有几个驱动时，main 函数里依次调 init。代码长这样：

```c
int main(void)
{
	clock_init();
	gpio_init();
	led_init(&red_led, 13);
	led_init(&green_led, 14);
	app_main();
}
```

3 个驱动以内这样写没问题。10 个以上 main 开始变得难读、容易漏改、依赖顺序靠注释维护。

**阶段二：手动分级 + 函数指针数组。** 工程师意识到分组的价值，开始这样写：

```c
typedef void (*init_fn_t)(void);

static init_fn_t board_inits[] = { clock_init, gpio_init, NULL };
static init_fn_t device_inits[] = { led_init_all, eeprom_init, NULL };
static init_fn_t app_inits[] = { app_state_machine_init, NULL };

int main(void)
{
	for (int i = 0; board_inits[i]; i++) board_inits[i]();
	for (int i = 0; device_inits[i]; i++) device_inits[i]();
	for (int i = 0; app_inits[i]; i++) app_inits[i]();
}
```

进步在于"分组 + 顺序"被显式表达。问题在于：每加一个驱动要改这份数组（违反开闭原则），且数组定义和驱动实现在不同文件，新人很难找。

**阶段三：链接器收集 + 7 级分组。** 借助 GCC `__attribute__((section(...)))` 和链接器的"段合并"能力，让每个驱动**自己声明**自己属于哪一级，main 一行不写：

```c
INIT_ENV_EXPORT(led_cfg);          /* led 驱动自带这一行 */
INIT_ENV_EXPORT(temp_sensor_cfg);  /* 温度传感器自带这一行 */
```

链接器在编译期把所有 `INIT_ENV_EXPORT` 函数收集到 `moduleExport5` 段，运行时遍历这个段。这就是 Linux 内核、RT-Thread、各家工业项目 2010 年之后走的路。

为什么要走完这三阶段才到工业版？因为前两个阶段的痛只有写过的人才理解。直接给新人讲 "用 `__attribute__((section))`" 他听不懂为什么要这样绕。第 17 章用了一整章铺垫，才把这条路走通。

### 19.6.2 如果某一级有 10 个驱动，调用顺序谁决定？

这是 initcall 机制最容易问的问题。答案是：**链接器决定**。

`INIT_ENV_EXPORT(led_cfg)` 把一份函数指针放进 `moduleExport5` 段。同样 `INIT_ENV_EXPORT(temp_sensor_cfg)` 也放进同一段。两份函数指针在段里的相对顺序，取决于链接器处理 .o 文件的顺序，通常是 Makefile 里 `OBJS` 变量的列出顺序。

也就是说，**同一级内部的顺序你不能强保证**。你只能保证 BOARD 在 PREV 之前、PREV 在 DEVICE 之前。

实际工程里这不是大问题：同一级的驱动通常是平行的（不互相依赖）。如果同一级的两个驱动有先后依赖（比如 LED 必须在蜂鸣器之前），那就把它们拆到不同级，或者在 cfg 里手动调。

Linux 内核里也有同款问题，处理方式：在级别名后面加 `_sync` 子级（`device_initcall_sync`），或者通过 `late_initcall` 把那种"必须最后跑"的事情挂到第 7 级。主控板项目目前没有这种细分，是因为依赖关系简单到 7 级就足够。

### 19.6.3 第 8 级独立段：单元测试

工业代码里 `platform_module_export.h` 实际比正文讲的多挂一个级别：`UNIT_TEST_EXPORT(fn)` 编号 "8"，专门收单元测试函数。它**不被** `platform_module_export_exec()` 调起，所以系统启动期不会跑测试。生产固件里这一段就是不会触发的死代码。

调用方要跑单元测试时（PC mock 回归 / 测试固件 / 工厂下线检测），手动调一次 `platform_unit_test_exec()`，它把第 8 级整段函数指针 walk 一遍。日常 1-7 级和测试段 8 严格隔离：前者每次上电都跑，后者只在测试场景跑。

为什么不让 `platform_module_export_exec` 把第 8 级一起调起来？因为单元测试和"启动期跑驱动"语义完全不同。生产固件绝不该跑测试代码副作用。这一条本身没新概念，只是把同一套链接器收集 + 段遍历按使用场景分了第二组段。读 `industrial/platform_layer/platform_module_export.h` 看到 `UNIT_TEST_EXPORT` 不必奇怪即可。

## 19.7 中间件：主控板用的几个开源库

主控板项目里的 driver 不是孤立的。每个 driver 底下都用到几个中间件库吸收掉重复劳动。这一节列出最重要的几个，让你知道工业项目的中间件栈大概长什么样。

**lwrb（lightweight ring buffer）。** 轻量级环形缓冲区。用在 ADC 长时间采样场景：定时器中断里把 ADC 读数推进环形缓冲，应用层定期取走分析。固定大小，写满覆盖最旧。比手写 `float buf[N]; int idx;` 多两个能力：原子操作（无锁单生产单消费）、写满策略可配。这一类库 GitHub 上几十个，核心代码 200 行左右，工业项目随便选一个稳定版本扔进 `third_party/` 就用。

**Letter Shell（字母 shell）。** 嵌入式命令行解释器。在 UART 上敲 `temp_get` 能立刻打印当前温度，敲 `led_on green` 能立刻点绿灯。开发期方便联调，量产期用 `#ifdef` 关掉。比自己写 UART 接收 + 命令解析省几百行代码。用法是把 shell 命令注册成宏：

```c
static void cmd_temp_get(int argc, char *argv[])
{
	(void)argc; (void)argv;
	float t = 0;
	temp_sensor_read(temp_board, &t);
	printf("temp = %.2f C\n", t);
}
SHELL_EXPORT_CMD(SHELL_CMD_PERMISSION(0), temp_get,
                 cmd_temp_get, "get board temperature");
```

注意这个 `SHELL_EXPORT_CMD` 宏：它和 `INIT_ENV_EXPORT` 是同一个机制（链接器收集段）。Letter Shell 把所有命令塞进 `shell_cmd` 段，shell 启动时 walk 一遍这个段就拿到所有命令的列表。**OOP 见过的所有抽象，跑一辈子嵌入式都在反复见同样的几招**。

**EasyLogger（elog）。** 嵌入式日志系统。`log_d / log_i / log_w / log_e` 四级日志，输出到 UART 或 RAM 或 Flash。比 `printf` 多三个能力：分级过滤（运行期改级别）、模块标签（每个文件 `#define LOG_TAG "led_cfg"`）、异步写出（写日志不阻塞业务线程）。前面 19.1 节 / 19.3 节配置代码里的 `log_d / log_e` 就是这个库。

**MultiButton。** 按键去抖与事件库。把硬件按键的电平变化抽成"按下 / 抬起 / 长按 / 双击"事件。第 20 章会看到子控板把电机故障检测引脚也当作 button 用，硬件电平变化就是一种按键事件。这是教学版从来想不到的复用。

这些中间件加起来大概占主控板代码量的 15%~20%，覆盖"重复劳动 80%"。**工业项目和教学项目的另一个差距，就是工业项目敢用第三方库**。教学版总倾向"什么都自己写"，工业版的态度是"凡是有成熟开源库的事，自己写就是浪费时间和给项目埋 bug"。

中间件不是这本书的主题，这一节只是让你看见它存在。每个具体库的细节，附录 D 会列入参考资料。

## 19.8 教学版到工业版的 12 个差距

把整章看下来，从 ch01 教学版的 `struct led + led_on` 走到工业版主控板，加进来的工程纪律可以列成下面这张表：

| # | 维度 | 教学版 | 工业版 |
|---|---|---|---|
| 1 | 错误处理 | `return -1` | `platform_err_t` 含 `EINVAL/EIO/ENOSYS/ETIMEOUT` 等 10+ 种 |
| 2 | NULL 校验 | `if (!me) return -1;` | 多层级深度校验（dev、ops、ops 表里的虚方法） |
| 3 | 总线访问 | 直接 `platform_pin_write` | 经 `platform_i2c_bus_device_find` 查表后 `platform_i2c_transfer` |
| 4 | 引脚标识 | `uint8_t pin` | 字符串 `"PI.15"` 经 `platform_pin_get` 查表 |
| 5 | 句柄类型 | `extern struct led red_led;` | `extern struct led_base *red_led;` 基类指针 |
| 6 | 实例创建位置 | `main()` 里全局变量 | 独立 `*_cfg.c` 文件里 `static` 实例 |
| 7 | 启动 | `main()` 手动调 init | INIT_xxx_EXPORT 宏，链接器自动收集 |
| 8 | 启动顺序 | 靠程序员记忆 | 7 级分组（BOARD → PREV → DEVICE → COMPONENT → ENV → APP → READY） |
| 9 | 多 callback | 单 cb 或没有 | 多事件独立 cb（风扇状态变化 / EEPROM 读写错） |
| 10 | 数据解码 | 简单 0/1 | 16 位定点数解析、二的补码、跨页拆分 |
| 11 | 中间件 | 全部自己写 | lwrb / Letter Shell / EasyLogger / MultiButton 等成熟库 |
| 12 | 设备发现 | hardcoded 全局 | `platform_device_register/find` 注册表（按字符串名查找） |

这 12 件事单独看每件都不复杂，挤在一份代码里就是工业级和教学级的全部差距。看懂的关键：每一件都不是凭空多出来的复杂度，都对应一个具体的工程问题。

读这一章读到这里，你应该有一种感觉：**这些代码每一行你都能 follow**。差距不在概念，差距只在"工业项目要面对的问题更多"。这就是这本书的目的。

### 19.9.1 LED 驱动（19.1 节）

### 19.9.2 EEPROM 驱动（19.2 节）

19.2 节贴的是 `eeprom_base.h` 接口骨架 + 应用层使用模式。**完整 `.c` 实现（`eeprom_ds2433.c` / `eeprom_i2c.c`）没有放进本仓库**。原因是 DS2433 1-Wire 协议实现涉及大量时序细节，这些细节是芯片驱动级的工程问题，跟本书的 OOP 教学主线关系不大。

19.2 节正文已经把 OOP 角度想讲的事讲完：

- 基类只放 ops 字段
- ops 表的 `read / write` 接口为什么返回 `int32_t`（POSIX 风格）
- 跨页 / 边界 / 超时三种"短读短写"场景为什么要返回实际字节数
- 应用层判断 `n < 0`（错误） / `n < size`（部分成功） / `n == size`（完整成功）的标准模式

这些信息足以让你拿任何一颗 EEPROM 芯片自己写一份子类实现。

### 19.9.4 风扇驱动（19.4 节）

19.4 节贴的是 `fan.h` 接口 + 应用层注册回调样例 + 内部 `fan_status_update_internal` 函数。**完整 `fan.c` 实现没有放进本仓库**。原因和 EEPROM 类似：UART 串口控制智能风扇的协议解析是芯片驱动级细节，对 OOP 教学价值有限。

19.4 节正文已经把 OOP 角度的全部要点讲完：
