# 第 7 章 · 写死的函数怎么换

配套代码：[`oop-in-c/code/07-function-pointer/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/07-function-pointer/)

## 7.1 一个真实场景

第 6 章你把 `name + is_on` 提到了 `struct led_base`，子类的 init 第一行调基类 init。继承的两层意义都齐了：数据共享、行为共享。

但有些行为它继承不了。比如 `led_on`。

红灯接 GPIO，开灯就是把引脚拉高。蓝灯接 PWM，开灯是配占空比。绿灯走 I2C 总线，开灯是发一条命令到地址 `0x50`。

三种 LED 的"开灯"逻辑完全不同。你写的 `led_on()` 函数体里到底要不要写 `platform_gpio_write`？写了它，蓝灯不对；不写它，红灯不亮。

朴素的想法是写三个函数 `led_on_gpio / led_on_pwm / led_on_i2c`，应用层调谁取决于自己用的是哪种 LED。这就回到了 ch01 那种"加一种 LED 抄一遍"的老路。

应用层应该一直只调 `led_on(&xxx)`。底下走哪一支不归它操心。

## 7.2 朴素方案：写一个 if

最直觉的写法是给 LED 加一个 mode 字段，把所有可能的硬件参数全塞进去：

```c
struct led_universal {
	struct led_base base;
	uint8_t mode;       /* 0=GPIO, 1=PWM, 2=I2C */
	uint8_t pin;        /* GPIO 用 */
	uint8_t channel;    /* PWM 用 */
	uint8_t i2c_addr;   /* I2C 用 */
};

int led_on(struct led_universal *me)
{
	if (me->mode == 0) {
		platform_gpio_write(me->pin, true);
	} else if (me->mode == 1) {
		platform_pwm_set(me->channel, 100);
	} else if (me->mode == 2) {
		platform_i2c_write(me->i2c_addr, 0x01);
	}
	return 0;
}
```

能跑。但有问题。

加第四种 LED（比如 SPI 控制的 LED 矩阵），你要改两处：定义一个新的 mode 数字，往 `led_on` 里加一个 else if。`led_off`、`led_set_brightness` 同样要改。改的还都是 led.c 里的代码（驱动核心），出 bug 影响所有 LED。

加一种 LED 改三个核心函数，这就是绑死。改一个 mode 数字定错，所有 LED 全乱。

问题在哪？`led_on` 里写死了 mode 和实现的对应关系。这个对应关系不应该由 `led_on` 决定。

## 7.3 把"调谁"做成可填的字段

冷静下来问一个问题：`led_on` 函数体里那一行 `platform_xxx_write` 到底是什么？

它是一段代码。每段代码编译完成后，都有一个内存地址。`platform_gpio_write` 函数有它自己的地址，`platform_pwm_set` 也有自己的地址。

地址是个数字，数字能存在变量里。

那能不能在每种 LED 子类里加一个字段，用来存"开灯这段代码的地址"？

```c
struct led_gpio {
	struct led_base base;
	uint8_t pin;
	int (*on_func)(struct led_gpio *me);   /* 存"开灯怎么做"的地址 */
};

struct led_pwm {
	struct led_base base;
	uint8_t channel;
	uint8_t duty;
	int (*on_func)(struct led_pwm *me);    /* PWM 子类自己的 */
};
```

`int (*on_func)(struct led_gpio *me)` 这一行的意思是：`on_func` 是个变量，它存的是一个函数的地址。这个函数接受一个 `struct led_gpio *` 参数，返回 `int`。

GPIO LED 创建的时候填 `gpio_on_pull_high` 这个函数的地址。PWM LED 创建的时候填 `pwm_on_full_duty`。

`led_on()` 自己长这样（按子类分别写一份；ch11 会演化为统一接口）：

```c
int led_gpio_on(struct led_gpio *me)
{
	if (!me || !me->on_func)
		return -1;
	return me->on_func(me);
}
```

调谁？me 自己说。`led_gpio_on` 这个函数从此一行不改。

## 7.4 函数也有地址

变量有地址你已经习惯了。`int a;` 这个变量在内存里某个位置躺着，`&a` 取它的地址。

函数呢？函数编译完是一段机器码，存在程序的代码段（text section）里。一段机器码的起点也是个地址。`gpio_write` 这个函数有自己的起点地址，可能是 `0x08001234`，`pwm_set` 在另一个地址 `0x08002A00`。

C 语言里，函数名出现在表达式里时，会自动退化成"这个函数的地址"（C99 标准 6.3.2.1 节第 4 段）。所以：

```c
red_led.on_func = gpio_on_pull_high;   /* 函数名不带括号, 取地址 */
red_led.on_func(&red_led);             /* 函数名加括号, 拨号执行 */
```

不带括号是号码（地址），加括号是拨号（执行）。和你给朋友发短信存号码、然后按通话键拨号是一样的两步。

`&gpio_on_pull_high` 也合法（显式取地址），`(*red_led.on_func)(&red_led)` 也合法（显式解引用），但 C 标准允许你两边都省略 `&` 和 `*`，所以工业代码里几乎都是简洁版。

## 7.5 换号码 vs 改实现

旧写法（mode + if-else）：加一种 LED，改 `led_on / led_off / led_set_brightness` 三个核心函数。

新写法（函数指针字段）：加一种 GPIO LED 子型号（比如有的拉高点亮、有的拉低点亮），写一个新的 `int gpio_on_xxx_style(struct led_gpio *me)`，创建实例时把它填进 on_func 字段。`led_gpio_on` 这个函数一行不改。

```c
int gpio_on_pull_low(struct led_gpio *me)   /* 低有效，新加 */
{
	platform_gpio_write(me->pin, false);
	me->base.is_on = true;
	return 0;
}

struct led_gpio matrix_led;
led_gpio_init(&matrix_led, "matrix", 8, gpio_on_pull_low);  /* 填进去就完事 */
```

驱动核心代码（`led_gpio_on` 内部）从此和"具体走哪一支"完全解耦。不管你加几种点亮电平，`led_gpio_on` 一行不改。

这就是软件工程里反复念的那句话："对扩展开放，对修改关闭。"加新功能不需要改老代码。`led_gpio_on` 是老代码，它对修改关闭；通过填一个新函数指针，行为对扩展开放。

## 7.6 这个东西叫什么

你刚才做的事：给 struct 加一个变量，这个变量存某个函数的地址，需要的时候通过这个变量调用函数。

软件工程里有个名字。它叫**函数指针**（function pointer）。

C++ 里这件事一字不改：

```cpp
class led_gpio {
public:
	int (*on_func_)(led_gpio *me);    /* 一模一样的函数指针字段 */
};
```

C++ 的 lambda、`std::function`、`virtual` 函数底层都是函数指针。C++ 给你包了几层语法糖，让写法更简洁，但骨头没换。你这一章写的就是 C++ 虚函数底层做的事。

第 11 章你会看到，把"对每个对象绑一组函数指针"这件事系统化，就是面向对象的多态。

## 7.7 视频里没讲透的几个细节

### 7.7.1 函数指针变量到底占几个字节

`sizeof(int (*)(struct led *))` 在 32 位 ARM Cortex-M 上是 4 字节，在 64 位 PC / ARM 上是 8 字节。和数据指针一样大。

但有一个坑：C 标准没有保证函数指针和数据指针大小相同。C99 标准 6.3.2.3 节把 "function pointer ↔ object pointer" 强转列为未定义行为。在常见架构（ARM / x86 / RISC-V）上两者大小一致，跨架构移植代码也都这样假设。但有些 Harvard 架构（早期 8051、AVR）上代码段和数据段地址空间分开，函数指针可能比数据指针大或小。

工业代码里如果遇到 8051 这种古董，要查 datasheet 决定怎么处理。在主流嵌入式平台（STM32、ESP32、Linux SBC）上不用纠结。

### 7.7.2 调用一次函数指针的代价（直接 vs 间接）

直接调用一个普通函数（编译期就知道跳哪）：

```
BL  led_on_gpio_style       ; 直接跳转到链接期已知地址 (3 cycle)
```

间接调用（运行时才知道跳哪，函数指针存在变量里）：

```
LDR  r3, [r4, #offset]      ; 从 me->on_func 读地址 (3 cycle, 命中 D-cache)
BLX  r3                     ; 间接跳转 + 写 Link Register (3 cycle)
```

差别两件事：

1. **多一条 LDR**：要先把函数地址从内存 load 到寄存器
2. **跳转代价不同**：直接 BL 编译期 ROM 化，CPU 可以推测执行（branch prediction）；间接 BLX 必须等 r3 的值出来才知道跳哪，对 CPU pipeline 不友好

ARM Cortex-M4（无分支预测器）上两者差距小，约 1-2 个周期。但 Cortex-A 系列（Linux 跑的那种）有完整 branch prediction，间接调用如果 BTB（Branch Target Buffer）没命中，损失可能达到 10+ 周期。

实测：ARM Cortex-M4 @ 168MHz 一次间接调用约 28 ns，168 MHz 下大约 5 个时钟周期。日常驱动调用频率（开关灯每秒几十次到几百次）这点开销完全可以忽略。但要避开几个雷区：

- **中断处理函数（ISR）的关键路径**：每个 ns 都要算
- **PWM 高频更新**：微秒级时序，间接调用的 jitter 会影响波形质量
- **超低功耗 MCU 的 hot loop**：M0 没有分支预测，每次间接调用是固定开销，循环 1000 次就是 1000 倍

C++ virtual 函数调用就是这个开销。Stroustrup 那句"零成本抽象"在虚函数这里要打个小折扣（间接跳转 + 编译器没法 inline 虚调用），但这是工程上完全可接受的代价。Linux 内核里 fast-path（中断 + softirq）大量避开虚调用走直接函数，但 slow-path（设备驱动注册、open/read/write 慢路径）几乎全是 ops 表 dispatch。这个分界线是 Linus 反复强调的。

### 7.7.2.1 函数指针 vs 普通函数：反汇编对比

两段 C 代码：

```c
/* 直接调用 */
int led_on_direct(struct led *me)
{
	return led_on_gpio_style(me);
}

/* 间接调用 */
int led_on_indirect(struct led *me)
{
	return me->on_func(me);
}
```

godbolt 上用 `arm-none-eabi-gcc -O2` 编译，分别得到（简化）：

```
led_on_direct:
	B   led_on_gpio_style       ; 一条 tail-call branch
led_on_indirect:
	LDR r3, [r0, #4]            ; r3 = me->on_func
	BX  r3                      ; tail-call jump
```

`led_on_direct` 在链接期就知道跳哪，编译器直接生成一条无条件跳转；`led_on_indirect` 必须先 load 才能跳。这就是"运行时绑定"在汇编层面的具体形态。

### 7.7.3 NULL 函数指针检查必须做

```c
int led_on(struct led *me)
{
	if (!me || !me->on_func)
		return -1;
	return me->on_func(me);
}
```

如果 `me->on_func == NULL` 直接调用，就是"跳到地址 0 执行"。在 ARM Cortex-M 上地址 0 是向量表起点（一段非代码数据），跳过去执行的不是代码，结果不是 HardFault 就是跑飞。在 Linux 用户态收到 SIGSEGV，core dump。

工业代码里所有函数指针调用前都要 NULL check。第 14 章会专门讲"函数指针没填"这件事的三种处理策略：报错、空实现、Linux 内核风格的 stub 函数。

### 7.7.4 函数指针字段在 struct 里的位置

`on_func` 现在放在 `struct led_gpio` 的最后。这个位置不影响功能，但影响内存布局（32 位 ARM）：

```
struct led_gpio {
	struct led_base base;              /* offset 0, 8 bytes (name 4 + is_on 1 + pad 3) */
	uint8_t pin;                       /* offset 8, 1 byte */
	/* offset 9, 3 bytes padding */
	int (*on_func)(struct led_gpio *); /* offset 12, 4 bytes */
};                                     /* sizeof = 16 */
```

函数指针的对齐是机器字长（4 或 8 字节），所以前面 1 字节字段后会有 padding。如果你在 RAM 紧张的 MCU 上想压榨内存，可以把对齐大的字段往前挪：

```c
struct led_gpio {
	int (*on_func)(struct led_gpio *); /* offset 0, 4 bytes (对齐 4) */
	struct led_base base;              /* offset 4, 8 bytes */
	uint8_t pin;                       /* offset 12, 1 byte */
	/* offset 13, 3 bytes padding */
};                                     /* sizeof = 16, 一样 */
```

这里 sizeof 没省。但如果再多两个 1 字节字段，调整后能省 8 字节。

但代价是 base 不在第一个位置了。基类向上转型就崩。所以工业代码里 base 永远第一，函数指针在后面。这是性能和向上转型机制之间的取舍，向上转型机制赢。

### 7.7.5 为什么用 typedef 简化函数指针类型

`int (*on_func)(struct led_gpio *me)` 写起来有点丑。如果 struct 里有 5 个函数指针：

```c
struct led_gpio {
	struct led_base base;
	uint8_t pin;
	int (*on_func)(struct led_gpio *);
	int (*off_func)(struct led_gpio *);
	int (*toggle_func)(struct led_gpio *);
	int (*set_brightness_func)(struct led_gpio *, uint8_t);
	int (*get_state_func)(struct led_gpio *);
};
```

读起来累。第 9 章会引入 typedef 给函数指针起短名字：

```c
typedef int (*led_action_fn)(struct led_base *me);
```

然后 ops 表里写：

```c
struct led_ops {
	led_action_fn on;
	led_action_fn off;
	led_action_fn toggle;
};
```

整个一章都讲这件事，所以本章先不用 typedef。

### 7.7.6 函数指针存在哪：text 段 vs 数据段

C 程序编译后大致分这几段（ELF 可执行格式）：

| 段 | 内容 | 权限 |
|---|---|---|
| `.text` | 函数机器码 | r-x（可读可执行不可写） |
| `.rodata` | 常量数据，比如 const 全局对象 | r--（只读） |
| `.data` | 已初始化的可变全局变量 | rw-（可读写） |
| `.bss` | 未初始化的全局变量 | rw-（运行时清零） |

`gpio_on_pull_high` 这个函数的机器码躺在 `.text` 段，地址在链接期就定了（在 STM32 上一般是 `0x08001234` 这种 Flash 地址）。

`me->on_func` 这个字段是 `struct led_gpio` 实例的一部分，住在 `.data` / `.bss` / 栈或堆里（取决于这个 led 怎么定义的）。

把函数地址赋给字段：

```c
red_led.on_func = gpio_on_pull_high;
```

这一行等价于把 `.text` 里某段代码的起点地址（一个常量）写进 `.data` 段（如果 `red_led` 是全局变量）。运行时调用 `me->on_func(me)` 就是读字段、跳到 text 段、执行那段机器码。

ARM Cortex-M 上还有一个细节：函数地址的最低位是 **Thumb 位**。Thumb 指令集的函数地址都加 1（最低位置 1），处理器根据这一位决定切到 ARM 还是 Thumb 模式执行。所以 `printf("%p", gpio_on_pull_high)` 你看到的可能是 `0x08001235` 而不是 `0x08001234`，最低那个 `5` 不是地址错位，是 Thumb 标志。`BLX` 指令会自动剥离这一位再跳。这是 ABI 规定，不是你的 bug。

### 7.7.7 ABI：函数指针调用如何传参

ARM EABI（Embedded Application Binary Interface）规定参数怎么传：

| 参数槽 | 寄存器 | 备注 |
|---|---|---|
| 第 1 个 | r0 | `me` 走这里 |
| 第 2 个 | r1 | brightness 等 |
| 第 3 个 | r2 | |
| 第 4 个 | r3 | |
| 第 5+ | 栈 | 按字对齐 |
| 返回值 | r0 | int / 指针 |

间接调用 `me->on_func(me)` 编译出来：

```
LDR  r3, [r0, #4]    ; r3 = me->on_func, 假设 on_func 在偏移 4
                     ; 这一步 r0 还保持是 me, 不能动
BLX  r3              ; 间接调用. 跳过去时 r0 仍然是 me, 等价于 on_func(me)
```

注意一件事：`r0` 在间接调用前就已经是 `me`（参数已经在位），只用 `r3` 读地址。所以不需要额外 `mov r0, r0`。这是 EABI 让间接调用代价小的关键设计。

如果你写 `me->on_func(other_obj)`（传别的指针），编译器会先 `mov r0, other_obj` 再 BLX，多一条指令。

### 7.7.8 函数指针的"丑语法"为什么是这样

`int (*on_func)(struct led *me)` 这个声明的读法是：

1. 看到 `*on_func`：on_func 是个指针
2. `(*on_func)(struct led *me)`：on_func 是指向"接受 struct led * 参数的"的指针
3. `int (*on_func)(...)`：on_func 是指向"返回 int、接受 struct led * 参数的函数"的指针

读法是从内向外。这个语法 1972 年 Dennis Ritchie 定的，那时候连结构化编程都还没占主流。后来加了 typedef 给它续命。

C99 没能改这个语法，因为改了会破坏巨量历史代码。C++11 加了 lambda + `auto` 让你能绕开它。Rust 用 `fn(...)` 类型，干净多了。

C 的丑保留下来不是因为它好，是因为它来不及改。

## 7.8 你现在的代码在 STM32 上长什么样

STM32 端胶水还是 ch01 那套（节选自 [`oop-in-c/code/07-function-pointer/stm32-snippet/led_stm32.c`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/07-function-pointer/stm32-snippet/led_stm32.c)）：

```c
void platform_gpio_write(uint8_t pin, bool value)
{
	HAL_GPIO_WritePin(GPIOA, (uint16_t)(1U << pin),
	                  value ? GPIO_PIN_SET : GPIO_PIN_RESET);
}
```

`led.h / led.c / main.c` 一字不改。函数指针字段加在 `struct led` 里，所有平台胶水都看不见这件事。

一个真实场景：同一颗 LED 在调试模式下用 gpio_style（按下按钮立刻亮），量产模式下用 pwm_style（按 brightness 调光）。出厂烧录时填哪个 on_func 决定行为，不需要重新编译 led.c。

本节用的还是函数式包装的 platform 抽象层，是教学简化版。第 11 章后会演化成 ops 表式。

## 7.9 你现在的代码在 Linux 用户态长什么样

Linux 端 sysfs 实现（节选自 [`oop-in-c/code/07-function-pointer/linux-snippet/led_linux.c`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/07-function-pointer/linux-snippet/led_linux.c)）：

```c
void platform_gpio_write(uint8_t pin, bool value)
{
	char path[64];
	int fd;

	snprintf(path, sizeof(path),
	         "/sys/class/gpio/gpio%u/value", (unsigned)pin);
	fd = open(path, O_WRONLY);
	if (fd >= 0) {
		write(fd, value ? "1" : "0", 1);
		close(fd);
	}
}
```

`led.h / led.c / main.c` 一字不改。同 7.8 节，platform 层是教学简化版，第 11 章后会改成 ops 表式。

## 7.10 工业代码里的函数指针字段

工业控制板项目里，每个驱动 struct 都有几个函数指针字段，用来给应用层注册"事件回调"。这一节只贴跟"函数指针字段"相关的部分，工业版 `led_base` 的完整字段集（含 `ops` 指针、`flags` 等）见 ch01 1.10 节、ch10 10.11 节、ch11 11.9 节：

```c
struct led_base {
	/* 跟回调相关的字段 (本节焦点) */
	const char *name;
	bool        is_on;
	void (*on_state_change)(struct led_base *me, bool new_state);
	/* ... 其他工业字段见 ch01 1.10 / ch10 10.11 / ch11 11.9 */
};
```

`on_state_change` 是一个回调字段。当 LED 状态变化时（比如温度报警触发自动闪烁），驱动会调用这个回调通知应用层。应用层注册的回调可以是"打日志"、"发到上层主控"，等等。

这种写法的本质就是你这一章学的：在 struct 里塞一个函数指针字段。下一章看怎么把函数指针当作参数从外面传进去（注册），把这个机制变成完整的回调系统。

你现在能把"调谁"塞进 struct 字段了。但有一个限制：必须先创建 struct 实例、再填 on_func，才能 led_on。如果有一个工具函数 `test_led(struct led *me)`，里面想调"开 → 等 → 关"三步，它该怎么知道用哪个 on？
