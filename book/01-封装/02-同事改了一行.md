# 第 2 章 · 同事改了一行 LED 全乱了

配套代码：[`oop-in-c/code/02-static-hiding/`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/02-static-hiding/)

## 2.1 一个真实场景

第 1 章你给三颗 LED 发了挂号单，代码只写一份，问题解决了。

提交、合并、上线。

一周后同事来找你：你 LED 模块好像有 bug，红灯不亮了。

你去看代码，main.c 里没动过。再翻 git log，发现同事在另一个文件里加了一行：

```c
extern struct led red_led;

/* 我做了点优化，让 pin 直接 = 999 */
red_led.pin = 999;
```

他不是恶意的。他可能在做硬件适配实验，可能手抖打错变量名，可能临时调试忘了撤回。但效果是确定的：你的红灯被驱动到一个不存在的引脚，从此不亮。

你打开微信想质问他，想了想，默默把聊天框关了。你心想：我的代码明明没问题啊。

对，你的代码确实没问题。问题是别人能碰你的代码。你的 struct 是敞开的，任何人从任何地方都能 `red_led.pin = ...` 把它改坏。

![你的数据是敞开的](../assets/ch02/slide1_问题代码.png)

银行不会把客户的钱放在路边让大家自取。柜台是有的，密码是有的，流程是有的。你的 struct 就是钱，函数就是柜台，现在缺一把锁。

## 2.2 第一把锁 · static

C 语言给你一把锁，叫 `static`。

去过饭店。前台是给客人用的（点菜、付钱、吃饭），后厨是给厨师用的（备料、炒菜）。客人进后厨只会添乱。所以后厨的门是锁的。

代码也一样。有些函数是给外面用的（开灯、关灯、调亮度），这些是前台。有些函数是你内部用的（更新硬件状态、检查参数合不合法），这些是后厨。

怎么把后厨锁起来？在函数前面加一个词 `static`。

加了 `static`，这个函数就只在它所在的 `.c` 文件里能用。别的 `.c` 文件想调它，链接器报 `undefined reference`。门锁上了，找都找不到。

变量也一样。文件作用域的变量加 `static`，就只在本文件可见。

![static = 后厨的门锁](../assets/ch02/slide2_static后厨.png)

举个例子。`led.c` 里有两个内部工具函数，一个统一更新硬件状态，一个检查亮度合不合法：

```c
static void update_hardware(struct led *me)
{
	platform_gpio_write(me->pin, me->is_on);
}

static bool brightness_valid(uint8_t brightness)
{
	return brightness <= 100;
}
```

前面加 `static`，这两个函数对 `main.c` 不存在。`main.c` 里写 `update_hardware(red);` 链接器直接报错。后厨锁上了。

## 2.3 第二把锁 · 头文件 = 菜单

锁后厨只解决一半问题。`static` 锁住的是函数，但你的 struct 字段 `pin / brightness / is_on` 还是敞着。同事 `red_led.pin = 999` 这一行还是能改坏你的数据。

得给客人发一份菜单。菜单上写什么？菜名和价格（也就是函数声明）。菜单上不写什么？配方（也就是字段定义和函数实现）。

你的 `.h` 文件就是菜单。`.c` 文件就是配方。

第 1 章 `led.h` 里写了完整的 struct 定义：

```c
/* ch01 led.h */
struct led {
	uint8_t pin;
	uint8_t brightness;
	bool    is_on;
};
```

`#include "led.h"` 的任何文件都看得到字段名、字段顺序、字段类型，自然也都能 `me->pin = 999` 直接改。

ch02 的修复很小：把字段定义从 `led.h` 搬到 `led.c`，`led.h` 里只留一行：

```c
/* ch02 led.h */
struct led;
```

这行叫前向声明（forward declaration）。它告诉编译器：有这么个类型，但具体长什么样我不告诉你。

字段名一字不动（`pin / brightness / is_on`），改的只是 visibility。

外部代码：

```c
#include "led.h"   /* 只看到 struct led; */

struct led *red = led_create(13);
red->pin = 999;       /* 编译报错: invalid use of undefined type */
```

`gcc` 在 `red->pin` 这一行直接拒绝继续：它知道有个 `struct led`，但不知道里面有没有 `pin`。这种"看得到指针，看不到字段"的写法叫不透明指针（opaque pointer）。

字段定义只在 `led.c` 内部：

```c
/* ch02 led.c */
struct led {
	uint8_t pin;
	uint8_t brightness;
	bool    is_on;
};
```

外部要操作 LED，只能走 `led.h` 暴露的函数：`led_create / led_destroy / led_on / led_off / led_set_brightness / led_get_state`。这些函数是菜单上写明的菜，外部能点。其他的，菜单上没有，碰不到。

![头文件 = 菜单](../assets/ch02/slide3_头文件菜单.png)

## 2.4 这个东西叫什么

你刚才跟我做的事，把不该让外面碰的东西藏起来，只通过接口和外界打交道，软件工程里有个名字。

它叫信息隐藏（information hiding）。

David Parnas 1972 年那篇 "On the Criteria To Be Used in Decomposing Systems into Modules" 第一次系统讲清楚这件事。中文叫信息隐藏，也常归到 encapsulation 这个大概念底下。

你不是从我这里背了一个定义，是从一个具体痛点（同事改一行你的 LED 全乱了）出发，自己推出了 `static` + 不透明指针这套解。这种被自己说服的理解，是背不出来的。

费曼讲过一句话：好的设计不靠人小心，靠机制上不让错误发生。飞机驾驶舱里关键开关都有保护盖，不是飞行员不靠谱，是一个手滑代价太大。你的代码也一样，锁起来不是不信任同事，是不让任何人手滑。

## 2.5 C 对 C++

如果你写过 C++，你会写：

```cpp
class Led {
public:
	int  on();
	int  off();
	int  set_brightness(uint8_t b);
private:
	uint8_t pin_;
	uint8_t brightness_;
	bool    is_on_;
	void    update_hardware();
};
```

而 C 里写的是：

```c
/* led.h */
struct led;
int led_on(struct led *me);
int led_off(struct led *me);
int led_set_brightness(struct led *me, uint8_t b);

/* led.c */
struct led {
	uint8_t pin;
	uint8_t brightness;
	bool    is_on;
};
static void update_hardware(struct led *me);
```

做的是一模一样的事。

C++ 的 `private:` 等同于 C 的"字段定义只放在 `.c`"加"工具函数加 `static`"。`public:` 等同于"声明放在 `.h`"。

底层机器码差不多。C++ 的 `private` 是编译期检查（编译器看到外部碰 `pin_` 直接报错），C 的不透明指针也是编译期检查（编译器看到外部碰 `me->pin` 直接报错）。运行时一行 cost 都没多。

![C 对比 C++](../assets/ch02/slide4_C对比Cpp.png)

第 1 章的恒等是 `struct + me = class + this`。

这一章的恒等是 `static + .h/.c 分离 = private + public`。

两条加起来，C++ 面向对象的两个核心机制（封装和成员函数），你已经全懂了。而且不是背的，是自己用 C 推出来的。

## 2.6 视频里没讲透的几个细节

### 2.6.1 不透明指针 vs 暴露字段，反事实推演

技术上你有三种藏字段的强度。

第一种，完全不藏（ch01 的写法）：字段在 `.h`，外部 `me->pin = 999` 编译过、链接过、运行也"过"，bug 在生产环境出现。

第二种，约定藏（注释里写"请勿直接修改"）：字段还在 `.h`，外部能改，靠 review 拦住。review 漏一次就是 bug。

第三种，机制藏（ch02 的写法）：字段在 `.c`，前向声明在 `.h`，外部 `me->pin = 999` 编译就过不去。0 漏网概率。

哪种是工业代码里的主流？看场景：

- **跨二进制库边界**用第三种。libc `FILE *` / POSIX `pthread_t` / Win32 `HANDLE` / RT-Thread `rt_device_t` / Linux 内核 `struct kobject *` 这些都是。应用代码和库内部物理分离，中间只能传句柄
- **同一份工程一起编译的应用代码**（你做的产品里大部分代码属于这种）几乎全用第二种：字段在 `.h` 公开 + `static` 工具函数 + 命名纪律。GitHub 上 nginx、Redis、LVGL 这一类主流 C 项目，struct 字段都在 `.h`，靠纪律隔离。第 4 章数据归位会把这种工业常态形态展开

第三种的代价是字段隐藏后调用方不能 `sizeof`，所有对象生命周期得由库自己管（你需要 `xxx_create / xxx_destroy` 这一对接口）。这在跨二进制库场景值得，因为 ABI 兼容是硬需求。在自家应用代码里，字段 `.h` 公开 + 直接 `struct led red;` 用对象本身，工程上更顺手。

工业控制板项目里 `eeprom_base_t` 是中间形态：

```c
/* eeprom_base.h */
struct eeprom_ops;            /* 前向声明 ops 表 */

typedef struct {
	struct eeprom_ops *ops;   /* 这个字段对应用公开 */
} eeprom_base_t;
```

`struct eeprom_ops` 字段（一堆函数指针）私有，但外层 `eeprom_base_t` 公开能 `sizeof`。这叫"部分藏"，藏 ops 表的内部接口，不藏外层壳。本书第 9 章会演化到这个形态。

ch02 主体演示的是第三种（完整不透明指针），是为了把"信息隐藏"机制讲透。后面章节大多用第二种 + 直接持有对象的工业常态写法。

### 2.6.2 为什么 led.h 里 forward decl 要写 struct led;

你可能见过这种写法：

```c
typedef struct led led_t;
```

效果几乎一样。但 Linux 内核明确反对 typedef struct，理由在第 1 章 1.7.3 节讲过：`typedef` 把"这是个结构体"这件事藏起来。读 `led_t *me` 你不知道它能不能 `.` 或 `->`，读 `struct led *me` 你立刻知道。

本书统一不 typedef struct，包括前向声明。`struct led;` 这一行同时承担两件事：声明类型存在 + 不暴露字段。

### 2.6.3 led_create / led_destroy 用 malloc 的取舍

ch01 里 `struct led red_led;` 是栈上对象，外部能直接定义。ch02 字段藏起来后，外部 `struct led red;` 编译器立刻报错 `storage size of 'red' isn't known`。编译器不知道 struct 多大，自然不知道分多少栈。

只能让 `led.c` 内部分配。最直观的方式是 `malloc`，外部拿 `struct led *` 这个指针句柄回去。`FILE *` / `pthread_t` / `HANDLE` / `SOCKET` 都是这个模式。

`malloc` 在嵌入式里以前有"堆碎片、确定性、实时性"的争议。**这是 1990 年代裸机 8051 / 小 RAM MCU 时代留下的 dogma，不是现代嵌入式现实**。FreeRTOS heap_4 / heap_5、RT-Thread 的 mp 和 heap、量产级状态机框架的事件池，都是动态内存机制，工业控制和汽车电子项目里大量在用。MISRA C:2012 amendment 1、DO-178C 都已经允许有约束的动态内存。

真正的工业纪律不是"不用 malloc"，是**生命周期管理**：

- 配对销毁：每个 alloc 一定有对应的 free 路径
- `goto cleanup`：异常退出分阶段释放（2.6.8 节展开）
- 引用计数：共享所有权用 `kref_get / kref_put`（Linux 内核标准）

只有少数特殊场景才优先用静态预分配：8051 / M0 + 16 KB 这种小 RAM 场景，或者中断热路径每秒几百万次 alloc 的极端情况。这些情况下选 `malloc` 还是静态分配是工程决策，不是教条。第 4 章数据归位会展示同一组数据怎么按场景选静态实例 / `malloc` / 对象池。

### 2.6.4 命名规范决策

`led_create` / `led_destroy` 的命名是有考虑的。

不叫 `led_new` / `led_delete`：和 C++ 关键字撞，读代码容易误认为 C++。

不叫 `led_alloc` / `led_free`：alloc 暗示纯内存分配，create / destroy 暗示构造 / 析构语义。

不叫 `led_open` / `led_close`：open / close 是 POSIX 文件接口的语义，留给 sysfs 用。

工业代码里 `led_base_t *` 这种句柄常用 `led_xxx_init(led_xxx_t *me, ...)` 命名，因为对象是栈上分配（实例字段公开给本驱动模块），`init` 是"对已分配的对象初始化"，不涉及堆分配。等到第 4 章数据归位讲完，你能看出"什么时候该 create / destroy、什么时候该 init / deinit"的判断标准。

### 2.6.5 链接器层面，static 和不加 static 的差别

`static` 改变的不是编译期可见性，是链接期符号表。

不加 `static` 的函数（external linkage）：编译器把函数名写进 `.o` 文件的全局符号表，链接器在合并多个 `.o` 时能看到。不同 `.o` 里的同名函数，链接器报 `multiple definition`。

加 `static` 的函数（internal linkage）：编译器要么完全不写符号表（被内联或调用图剪枝），要么写成本地符号（链接器跨 `.o` 看不到）。

```
$ nm led.o
0000000000000000 T led_create
0000000000000098 T led_destroy
0000000000000114 T led_on
0000000000000180 T led_off
00000000000001a0 t update_hardware
00000000000001f0 t brightness_valid
```

大写 `T` 是 external，小写 `t` 是 file-local。这就是为什么外部 `update_hardware(red);` 链接报 `undefined reference`：链接器扫所有 `.o` 的全局符号表都找不到 `update_hardware`。

### 2.6.6 不透明指针的运行时代价

零。

`red->is_on` 在 `led.c` 内部和 ch01 的 `red->is_on` 编译出同样的汇编：

```
LDRB  r1, [r0, #2]      ; 偏移 2 是 is_on 字段
```

`led_on(red)` 在 main.c 里编译出 `BL led_on`（branch with link，跳转并保存返回地址）。函数调用本身有几个周期开销，但这和 ch01 是一样的（ch01 也是函数调用）。

外部代码"看不见字段"是编译期机制，链接出二进制后字段还在那里、布局不变、偏移不变。Bjarne Stroustrup 那句"不用的特性零成本"在 C 上同样成立。

### 2.6.7 Linux 内核 struct file 也是一个不透明指针

把视角拉到工业级。Linux 内核的 `struct file`（每打开一个文件就有一份）就是不透明指针的活样本。

`include/linux/fs.h` 里只声明 `struct file`，里面的字段（`f_pos`、`f_flags`、`f_inode` 等几十个）只对 fs 子系统内部公开。其他子系统（drivers / net / sched）拿到的是 `struct file *`，能传它、能挂回调，但不能 `f->f_pos = 0` 直接改。

为什么内核要这么设计？

因为 `struct file` 的字段在不同内核版本之间会变（4.x 加了几个字段，5.x 把另几个字段挪进 `struct file_ra_state`）。如果 driver 直接读字段，每升一次内核所有 driver 都要改。藏起来后 driver 走 `f->f_op->read(f, ...)` 这种接口调用，字段怎么变都不影响。

这本书第 16 章会展开 `struct file` 和 `struct file_operations` 的工业级用法。这一章你只需要看到：你刚才在 `struct led` 上做的事，Linux 内核 4000 万行代码也在做，机制完全一样。

### 2.6.8 句柄生命周期 · C 语言的 RAII 类比

C++ 有 RAII（Resource Acquisition Is Initialization）：对象进入作用域自动构造、离开作用域自动析构。资源不会泄漏。

C 没有 RAII。`led_create` 拿到 `struct led *` 之后，谁负责 `led_destroy`？只能靠工程纪律。

工业代码里管这种"谁创建谁销毁"的纪律，最常见有三套写法。

**配对释放**：哪个函数 `create`，同一个调用者 `destroy`。简单直观，但函数中间 `return` 多了容易漏。这是 C 标准库 `fopen / fclose`、`malloc / free` 的写法。

**goto cleanup**：函数中间任何一步失败，`goto out` 集中走清理路径。Linux 内核风格，几十年实践证明能压住所有 leak。形态是：

```c
struct led *led_create_two(void)
{
	struct led *r = NULL, *g = NULL;

	r = led_create(13);
	if (!r)
		goto fail;

	g = led_create(14);
	if (!g)
		goto fail_g;

	return r;     /* 这里要把两个都返回，省略 */

fail_g:
	led_destroy(r);
fail:
	return NULL;
}
```

每一个 `goto` 都把已经分配的资源对应释放。从前往后看是分配链，从后往前看是清理链，对称。

**引用计数**：对象自己带一个 `refcount`，每个使用者 `get` 一次 `+1`，用完 `put` 一次 `-1`，归零自动 destroy。Linux 内核 `kref_init / kref_get / kref_put` 是标准模板，几乎所有内核对象（`struct device`、`struct file`、`struct kobject`）都用这套。本章先不展开，第 17 章讲完 initcall 后会回到这条线。

ch02 的 `led_create / led_destroy` 是配对释放风格。`led.c` 内部用 `malloc`，调用者拿 `struct led *` 句柄，结束时 `led_destroy` 负责 `free`。这是初学最干净的形态。

### 2.6.9 堆分配 vs 栈分配 · 按场景选

ch01 写 `struct led red_led;`（栈分配），ch02 改成 `struct led *red = led_create(13);`（堆分配）。两种风格各有各的代价。

| 维度 | 栈 / 静态实例 (`struct led red`) | 堆 (`led_create`) |
|---|---|---|
| 字段是否可见 | 必须可见（编译器要算 size） | 可隐藏（只暴露指针） |
| 分配开销 | 几乎零（移栈指针） | malloc 内部要找空闲块 |
| 释放开销 | 自动（出作用域） | 手动 free，漏一次就 leak |
| 实时性 | 确定（O(1)） | RTOS heap_4/5 有保证；newlib dlmalloc 不保证 |
| 内存占用 | 函数调用结束就回收 / 整个程序生命周期占着 | 在 free 前一直占着 |
| 适合场景 | 数量固定的全局对象（LED / 串口 / 传感器） | 生命周期不固定的对象（动态接收的数据包、临时缓冲区、按需创建的会话） |

工业 MCU 代码里两种都常用，按对象生命周期选。同一份工程里**全局少量对象**通常 `static struct xxx instance;` 或顶层模块持有，**生命周期不固定的对象**用 RTOS 提供的 heap（FreeRTOS heap_4/5、RT-Thread `rt_malloc`）。两种都是工业级常态。

ch02 里 `led_create + malloc` 是为了把"信息隐藏 + 句柄式 API"这件事讲清楚。隐式句柄风格存在于 `FILE *` / `pthread_t` / `rt_device_t` 这一类跨库边界场景。后面 ch04 数据归位会展开"实例字段公开 + 直接 `struct led red;` 静态实例"这种自家应用代码常态形态，跟 ch02 这种隐式句柄形态对照。两种是工程决策，不是非此即彼。

## 2.7 你现在的 LED 在 STM32 上长什么样

PC 模拟版是 `printf` 假装操作 GPIO。STM32 真实硬件上长这样（节选自 [`oop-in-c/code/02-static-hiding/stm32-snippet/led_stm32.c`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/02-static-hiding/stm32-snippet/led_stm32.c)）：

```c
#include "platform.h"
#include "stm32f4xx_hal.h"

void platform_gpio_init(uint8_t pin, uint8_t mode)
{
	GPIO_InitTypeDef cfg = {0};

	__HAL_RCC_GPIOA_CLK_ENABLE();

	cfg.Pin   = (uint16_t)(1U << pin);
	cfg.Mode  = (mode == GPIO_MODE_OUTPUT) ?
	            GPIO_MODE_OUTPUT_PP : GPIO_MODE_INPUT;
	cfg.Pull  = GPIO_NOPULL;
	cfg.Speed = GPIO_SPEED_FREQ_LOW;
	HAL_GPIO_Init(GPIOA, &cfg);
}

void platform_gpio_write(uint8_t pin, bool value)
{
	HAL_GPIO_WritePin(GPIOA, (uint16_t)(1U << pin),
	                  value ? GPIO_PIN_SET : GPIO_PIN_RESET);
}
```

`led.h` / `led.c` / `main.c` 一字不改。字段还藏在 `led.c` 里，`main.c` 还是看不到 `pin / brightness / is_on`。变化的只是这层 platform 胶水。

不透明指针在 ARM Cortex-M 上和 x86 上行为完全一致。它是编译期机制，编译器拒绝就拒绝，烧不到芯片上。

这一节用的是函数式包装的 platform 抽象，是教学简化版。真正工业级用虚函数表（ops 表），允许 runtime 切换平台。第 11 章演化出 vptr 之后，会把 platform 层重构成 ops 表式，和 2.9 节工业代码对齐。

## 2.8 你现在的 LED 在 Linux 用户态长什么样

嵌入式 Linux 端长这样（节选自 [`oop-in-c/code/02-static-hiding/linux-snippet/led_linux.c`](https://github.com/ZhaoChengBo/zhaoming-embedded/tree/master/oop-in-c/code/02-static-hiding/linux-snippet/led_linux.c)）：

```c
void platform_gpio_init(uint8_t pin, uint8_t mode)
{
	int fd = open("/sys/class/gpio/export", O_WRONLY);
	if (fd >= 0) {
		dprintf(fd, "%u", (unsigned)pin);
		close(fd);
	}

	char path[64];
	snprintf(path, sizeof(path),
	         "/sys/class/gpio/gpio%u/direction", (unsigned)pin);
	fd = open(path, O_WRONLY);
	if (fd >= 0) {
		const char *dir = (mode == GPIO_MODE_OUTPUT) ? "out" : "in";
		write(fd, dir, strlen(dir));
		close(fd);
	}
}

void platform_gpio_write(uint8_t pin, bool value)
{
	char path[64];
	snprintf(path, sizeof(path),
	         "/sys/class/gpio/gpio%u/value", (unsigned)pin);
	int fd = open(path, O_WRONLY);
	if (fd >= 0) {
		write(fd, value ? "1" : "0", 1);
		close(fd);
	}
}
```

Linux 把每个 GPIO 暴露成文件 `/sys/class/gpio/gpio13/value`。写 `'1'` 拉高，写 `'0'` 拉低。这种"把硬件当文件"的设计叫 sysfs。

`led.h` / `led.c` / `main.c` 还是一字不改。

这一节同样是函数式包装的教学简化形态，工业级用 ops 表。第 11 章之后会演化对齐。

## 2.9 工业代码里的 led 长什么样

我做的工业控制板项目里，LED 这一块的 `.h` 是这样：

```c
/* led_base.h - 应用层只看这一份 */
typedef struct led_base led_base_t;       /* 不透明前向声明 */

void led_on(led_base_t *me);
void led_off(led_base_t *me);
```

```c
/* led_base.c - 字段定义和实现都关在这里 */
struct led_base {
	const char *name;
	uint8_t     pin;
	bool        is_on;
	/* 真实工程还有几个字段, 这里省略 */
};

void led_on(led_base_t *me)
{
	/* ... 实际 GPIO 操作 ... */
}
```

应用调用：

```c
extern led_base_t *green_led;
extern led_base_t *red_led;

led_on(green_led);
led_off(red_led);
```

这和你这一章学的不透明指针骨架一模一样：把"实现细节"塞进 `.c`，`.h` 只暴露指针类型 + 操作函数。

> 真实工业代码的 `led_base` 还多一个 `ops` 字段（用来在 GPIO LED / PWM LED / I2C LED 之间运行时切换实现）。这一招叫**虚函数表**或者**ops 表**，是第 9 章到第 11 章的主题。这一章先把"信息隐藏"这一招学透就够，不用懂 ops 表。

到这里你应该能形成一个直觉：拿到一份陌生的工业代码，先看 `.h` 里暴露了什么。如果暴露的是完整的 struct 定义，是 ch01 阶段的代码，正在等一次 ch02 重构。如果只暴露前向声明 + 操作函数，是 ch02 标准的代码（具体到工业项目里，再往下挖通常就是 ops 表 + 子类继承，那是 ch06 到 ch11 的内容）。
